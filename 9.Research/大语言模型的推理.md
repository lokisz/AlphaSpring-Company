# Prompt: 目前的SOTA大模型推理, 主要分为几个大的步骤, 各自对硬件架构的需求有什么特点? 例如计算量, 存储带宽和容量等等
先审查前提：
你的问题把“推理”当成一个统一过程，这个前提不完全对。  
对当前 SOTA 大模型服务系统，**真正决定硬件需求的不是“模型参数量”本身，而是推理被拆成了哪些阶段，以及流量形态**：短 prompt / 长 prompt、短输出 / 长输出、dense / MoE、单轮 / agentic 多轮、是否复用前缀 KV cache。工程上最核心的分界是 **Prefill** 和 **Decode**，两者的硬件瓶颈几乎相反：前者更偏算力，后者更偏显存带宽与容量。vLLM、NVIDIA Dynamo、TensorRT-LLM 都围绕这个分界做了 chunked prefill、disaggregated prefill、KV cache 管理和 speculative decoding 等设计。([vLLM](https://docs.vllm.ai/en/v0.10.1/features/disagg_prefill.html?utm_source=chatgpt.com "Disaggregated Prefilling (experimental) - vLLM"))

## 一、SOTA 大模型推理的 4 个大步骤

### 1) 请求预处理与调度

包括 tokenizer、batching、请求排队、prefix cache 命中检查、路由到 prefill/decode engine。  
这一阶段通常**不是主算力瓶颈**，更多是：
- CPU 参与较多
- 对系统调度和软件栈要求高
- 对延迟敏感
- 对网络和跨节点元数据传递有要求

在现代 serving 系统里，调度器的重要性上升很快，因为 continuous batching、prefix cache、disaggregated serving 都依赖它决定 TTFT 和吞吐。NVIDIA Dynamo 明确把资源调度、内存管理、数据传输作为分布式推理框架核心能力；TensorRT-LLM 和 vLLM 也把 KV cache 与 batching 作为关键系统层优化。([NVIDIA Developer](https://developer.nvidia.com/dynamo?utm_source=chatgpt.com "Dynamo Inference Framework | NVIDIA Developer"))

### 2) Prefill（处理输入上下文）
把整段输入 prompt 一次性“读完”，完成所有层的前向计算，并生成这段上下文对应的 KV cache。 
这是用户发出请求后到首 token 出来前最重要的计算阶段。

**硬件特征：**
- **计算量高**
- **并行度高**
- 更接近 **compute-bound**
- 更吃 **矩阵乘吞吐、Tensor Core、低精度算力**
- 对显存带宽有要求，但通常不像 decode 那么极端
- 长 prompt 时显存写入压力大，因为要生成大量 KV cache

vLLM 文档直接把 chunked prefill 描述为在系统层帮助平衡 **compute-bound 的 prefill** 与 **memory-bound 的 decode**；多篇 roofline/survey 也把 prefill 视为高算术强度、偏算力受限阶段。([vLLM](https://docs.vllm.ai/en/v0.10.2/configuration/optimization.html?utm_source=chatgpt.com "Optimization and Tuning - vLLM"))

### 3) Decode（逐 token 自回归生成）
每次只生成一个或少数几个 token，重复执行很多步；每一步都要读模型权重，并访问不断增长的 KV cache。  
这是在线服务里最常见的主瓶颈阶段。

**硬件特征：**
- 单步计算量相对小，但步数多
- 算术强度低
- 通常是 **memory-bandwidth-bound**
- 极度依赖 **HBM 带宽**
- 对 **显存容量** 也敏感，因为 KV cache 会持续增长
- 对 **低延迟 kernel launch、调度、连续批处理** 很敏感

vLLM 的 speculative decoding 文档明确说它提升的是 **memory-bound LLM inference** 的 inter-token latency；Together 的长上下文分析也指出大 batch、长上下文下 decode 会因 KV cache 过大再次明显受内存限制。([vLLM](https://docs.vllm.ai/en/v0.8.5/features/spec_decode.html?utm_source=chatgpt.com "Speculative Decoding - vLLM"))

### 4) 后处理
包括 sampling、logits 处理、反 tokenization、格式约束、工具调用拼接等。  
通常不是 GPU 资源主瓶颈，但对端到端延迟仍有影响，尤其在小模型、短回答、强约束解码场景下会变得显著。TensorRT-LLM 把 sampling、guided decoding、logits processor 单列为推理功能模块，也说明这部分在系统实现里不能忽略。([NVIDIA GitHub](https://nvidia.github.io/TensorRT-LLM/?utm_source=chatgpt.com "Welcome to TensorRT LLM's Documentation!"))

## 二、每一步对硬件资源的需求画像

### A. 计算量（FLOPs / Tensor Core）
**最敏感阶段：Prefill**
原因：
- prompt 中所有 token 可以并行处理
- attention 和 MLP 的大矩阵乘能把 Tensor Core 吃满
- prompt 越长，prefill FLOPs 近似线性上升

因此 prefill 更偏好：
- 高 Tensor Core 吞吐
- FP8 / FP4 / INT8 等低精度高吞吐能力
- 更强的单卡算力密度

NVIDIA 在 2025–2026 年的 Blackwell / Rubin 叙述里，反复强调 Transformer Engine 和更低精度（如 FP8、NVFP4）对推理性能的价值，本质上就是在提升这类高吞吐矩阵计算阶段的效率。([NVIDIA Developer](https://developer.nvidia.com/blog/inside-nvidia-blackwell-ultra-the-chip-powering-the-ai-factory-era/?utm_source=chatgpt.com "Inside NVIDIA Blackwell Ultra: The Chip Powering the AI ..."))

### B. 显存带宽（HBM bandwidth）
**最敏感阶段：Decode**
原因：
- 每生成一个 token，都要重新读大量权重
- 还要读取历史 KV cache
- 每步计算不大，导致“每字节数据对应的计算”偏低，典型内存带宽瓶颈

所以 decode 更偏好：
- 更高 HBM 带宽
- 更低访存延迟
- 更好的 cache / SRAM / kernel fusion
- 更强的 continuous batching 与 speculative decoding

这也是为什么很多线上系统即便 FLOPs 很高，decode token/s 仍不理想：瓶颈不在算力峰值，而在“每一步需要从显存搬多少数据”。([vLLM](https://docs.vllm.ai/en/v0.8.5/features/spec_decode.html?utm_source=chatgpt.com "Speculative Decoding - vLLM"))

### C. 显存容量（HBM capacity）
**最敏感对象：模型权重 + KV cache**
NVIDIA 的推理优化材料直接指出，LLM 内存需求两大项就是：
1. **模型权重**
2. **KV cache** ([NVIDIA Developer](https://developer.nvidia.com/blog/mastering-llm-techniques-inference-optimization/?utm_source=chatgpt.com "Mastering LLM Techniques: Inference Optimization"))

其中：
- **权重容量** 决定模型能否整机驻留、是否需要张量并行/流水并行、是否要量化
- **KV cache 容量** 决定可支持的上下文长度、并发数、prefix cache 命中收益

一个非常实用的理解是：
- **短上下文**：显存主要被权重占用
- **长上下文/高并发**：KV cache 会快速成为第一大显存消耗项

NVIDIA 2025 年关于 KV cache reuse 的文章专门强调 KV cache 的增长与避免昂贵重算之间的平衡；Blackwell Ultra 也把“更大上下文长度、更多 KV cache 容量”作为核心卖点之一。([NVIDIA Developer](https://developer.nvidia.com/blog/introducing-new-kv-cache-reuse-optimizations-in-nvidia-tensorrt-llm/?utm_source=chatgpt.com "Introducing New KV Cache Reuse Optimizations in NVIDIA ..."))

### D. 节点间互连 / 网络带宽

这个维度在 **多卡、多机、MoE** 时变成一级问题。
#### Dense 模型
主要是：
- 张量并行的 all-reduce
- 流水并行的 stage 传递
- 多卡共享权重 / 激活

需求特点：
- 高带宽 NVLink / NVSwitch 明显优于普通 PCIe
- 多机时 InfiniBand / RDMA 很关键
- 低延迟互连对 decode 尤其重要
#### MoE 模型
MoE 额外增加 **专家路由的 all-to-all 通信**。  
Google Cloud 与 NVIDIA Dynamo 的 MoE 推理文章明确指出，DeepEP/WideEP 的核心就是加速 token 到专家的 dispatch / combine，也就是高带宽 all-to-all；这正是大规模 MoE 推理的常见同步瓶颈。([Google Cloud](https://cloud.google.com/blog/products/compute/scaling-moe-inference-with-nvidia-dynamo-on-google-cloud-a4x?utm_source=chatgpt.com "Scaling MoE inference with NVIDIA Dynamo on ..."))

结论：
- **Dense**：更像“权重切分 + 常规并行”
- **MoE**：在此基础上再叠加“频繁通信”
- 所以 MoE 往往对 **GPU 互连拓扑、交换带宽、调度器** 更敏感

## 三、按模型类型看，硬件瓶颈如何变化

### 1) Dense Transformer

最典型，阶段划分最清楚：
- Prefill：算力主导
- Decode：带宽主导
- 长上下文：KV cache 容量开始主导

因此 dense 模型的最佳硬件一般要求：
- 足够高的 Tensor Core 吞吐
- 足够大的 HBM 容量装下权重 + KV
- 足够高的 HBM 带宽保证 decode token/s

### 2) MoE 模型
MoE 不是“只看激活参数变少就更省硬件”。这个前提常常是错的。  
它通常能减少**每 token 实际参与计算的参数量**，但会引入新的系统代价：
- 专家分布到不同 GPU
- token 需要路由到目标专家
- all-to-all 通信与同步开销上升
- 专家负载不均会造成部分 GPU 空转

所以 MoE 的硬件画像是：
- 算力需求相对更稀疏
- **互连带宽/延迟** 更重要
- 调度与 expert placement 更重要
- 大规模集群拓扑更重要

Google Cloud + NVIDIA 在 2026 年的 MoE 推理案例中，把“最小化同步开销”“加速 dispatch / combine 高带宽 all-to-all”作为关键优化点，这基本就是 MoE 推理硬件需求的核心。([Google Cloud](https://cloud.google.com/blog/products/compute/scaling-moe-inference-with-nvidia-dynamo-on-google-cloud-a4x?utm_source=chatgpt.com "Scaling MoE inference with NVIDIA Dynamo on ..."))

### 3) 长上下文模型
长上下文不是简单多一点算力，而是会明显放大：
- prefill FLOPs
- KV cache 容量
- decode 访问 KV 的带宽压力

因此长上下文场景下，硬件常先撞到的是：
- HBM 容量
- HBM 带宽
- KV cache 管理能力

而不是单纯峰值 TFLOPS。关于这一点，长上下文推理和 KV cache 相关文章都强调了 KV cache 可能快速膨胀并压垮高带宽显存。([together.ai](https://www.together.ai/blog/speculative-decoding-for-high-throughput-long-context-inference?utm_source=chatgpt.com "Speculative decoding for high-throughput long-context ..."))

### 4) 推理型 / reasoning 模型

这里容易有一个误区：  
很多人以为“推理型模型”主要改变单步计算图。通常并不是。更常见的是：
- 输出 token 更多
- 思维链更长
- 多轮内部采样/验证更多
- test-time compute 更高

这意味着硬件上经常体现为：
- **decode 时间更长**
- KV cache 占用更久
- 并发更容易受挤压
- token 成本更高

所以 reasoning 模型未必最吃 prefill，反而常常更吃 **长期 decode 带宽、KV 容量、调度效率**。NVIDIA 在 2026 年公开材料里也把 Rubin/低成本 token 叙事和 agentic / advanced reasoning 直接绑定。([NVIDIA Newsroom](https://nvidianews.nvidia.com/news/rubin-platform-ai-supercomputer?utm_source=chatgpt.com "NVIDIA Kicks Off the Next Generation of AI With Rubin"))

## 四、为什么现在很多系统把 Prefill 和 Decode 分开部署

因为两者资源画像不同：
- **Prefill**：需要高算力、高并行、追求 TTFT
- **Decode**：需要高带宽、大 KV、追求 ITL / token/s

vLLM 和 NVIDIA Dynamo 都支持或强调 **disaggregated prefill**，就是把 prefill 与 decode 放到不同实例甚至不同 GPU 池，分别调优。vLLM 明确说这样可以独立优化 TTFT 和 ITL。([vLLM](https://docs.vllm.ai/en/v0.10.1/features/disagg_prefill.html?utm_source=chatgpt.com "Disaggregated Prefilling (experimental) - vLLM"))

这背后的硬件逻辑是：
- prefill 池可偏向“算力型 GPU”
- decode 池可偏向“带宽/显存型 GPU”
- 大流量下整体利用率更高
- 混合工作负载下尾延迟更稳

## 五、把硬件需求压缩成一句话

### 1) Prefill 看什么
看：
- Tensor Core 吞吐
- 低精度支持（FP8/FP4/INT8）
- 大 batch 并行能力
- prompt 长度下的稳定吞吐

### 2) Decode 看什么
看：
- HBM 带宽
- HBM 容量
- KV cache 管理
- 调度器与连续批处理
- speculative decoding / prefix reuse 支持

### 3) MoE 额外看什么
看：
- NVLink / NVSwitch / InfiniBand 带宽
- all-to-all 性能
- 专家并行与调度策略
- 通信与计算重叠能力

## 六、一个更准确的总结构图

可以把 SOTA 大模型推理硬件需求概括成：
**在线推理总瓶颈 = max( Prefill算力瓶颈, Decode带宽瓶颈, KV容量瓶颈, MoE通信瓶颈 )**
对应关系：
- **短 prompt、短回答** → 常偏 prefill 算力
- **短 prompt、长回答** → 常偏 decode 带宽
- **长 prompt、短回答** → prefill 算力 + KV 容量
- **长 prompt、长回答** → decode 带宽 + KV 容量
- **MoE 大模型** → 还要加上跨卡/跨机通信瓶颈

## 七、最实用的结论

### 结论 1
“推理最吃算力”只对了一半。  
对现代在线 LLM 服务，**decode 往往更受显存带宽和 KV cache 制约**，不是纯 FLOPs 制约。([vLLM](https://docs.vllm.ai/en/v0.8.5/features/spec_decode.html?utm_source=chatgpt.com "Speculative Decoding - vLLM"))
### 结论 2
“更长上下文”本质上是在同时放大三件事：
- prefill 计算
- KV cache 容量
- decode 访存压力

### 结论 3
“MoE 更便宜”也只对了一半。  
它可能降低每 token 有效计算量，但常把瓶颈转移到 **专家路由和 all-to-all 通信**。([Google Cloud](https://cloud.google.com/blog/products/compute/scaling-moe-inference-with-nvidia-dynamo-on-google-cloud-a4x?utm_source=chatgpt.com "Scaling MoE inference with NVIDIA Dynamo on ..."))

### 结论 4
SOTA serving 架构的方向，不是单纯堆更大 GPU，而是：
- prefill / decode 解耦
- 更强 KV cache 复用
- 更强 speculative decoding
- 更强 continuous batching
- 对 MoE 做专门通信优化

这些方向已经体现在 vLLM、TensorRT-LLM、NVIDIA Dynamo 的公开实现与文档里。([vLLM](https://docs.vllm.ai/en/v0.10.1/features/disagg_prefill.html?utm_source=chatgpt.com "Disaggregated Prefilling (experimental) - vLLM"))

## 八、你如果关心“买什么硬件”，优先看这四个指标

不要先看峰值 TFLOPS，先看：
1. **HBM 容量**：能装下多大权重和多大 KV
2. **HBM 带宽**：decode token/s 上限
3. **低精度推理能力**：prefill 吞吐
4. **GPU 互连/NVLink/网络**：多卡 dense 与 MoE 的真实效率

# Prompt: 详细描述推理的整个工作过程, 各个步骤的输入输出和中间结果, 以及对硬件计算/存储/网络的影响? 另外例如openclaw等agentic工具对推理的改变和硬件影响是? 

先纠正一个常见前提：**“推理”不等于“一次前向传播生成答案”。**  
对 today 的 SOTA 系统，完整推理链路通常是：
1. 请求进入与编排
2. 上下文构造
3. tokenization / embedding
4. prefill
5. KV cache 持久化/复用
6. decode 循环
7. sampling / 约束解码
8. detokenize / 格式化
9. 工具调用或环境交互（如果是 agent）
10. 将工具结果再次并入上下文，进入下一轮 prefill/decode

所以标准 chat 模([Reuters](https://www.reuters.com/business/media-telecom/baidu-joins-chinas-openclaw-frenzy-with-new-ai-agents-2026-03-17/?utm_source=chatgpt.com "Baidu joins China's OpenClaw frenzy with new AI agents"))“多轮 LLM 调用 + 外部工具/环境循环”**。OpenClaw 这类框架并不是改写 Transformer 内核，而是在外层给 LLM 加上 memory、tools、triggers、output channels，使其从单轮助手变成自治 agent。GitHub 和解读文章都把它定义为 personal AI assistant / agent harness；Reuters 报道也将其描述为可跨桌面、云、移动端执行多步任务的 agent 框架。([GitHub](https://github.com/openclaw/openclaw "GitHub - openclaw/openclaw: Your own personal AI assistant. Any OS. Any Platform. The lobster way.  · GitHub"))

# 一、 LLM 推理的完整工作过程
下面按**真实 serving 系统**拆解，而不是只按神经网络教材拆。
## 1) 请求进入与系统调度

### 输入
- 用户文本 / 多模态输入
- system prompt
- 历史对话
- sampling 参数
- 工具定义 / response schema / JSON schema（如有）

### 中间结果
- request object
- priority / queue metadata
- batch 分组信息
- prefix cache 命中检查结果
- 路由结果：去哪个 prefill/decode 实例

### 输出
- 待执行的“模型输入序列”
- 调度器上的 batch slot
- 可能命中的 prefix / KV blocks

### 硬件影响
**计算：**
- 主要在 CPU，GPU 占用很少
- 算力压力小，但对低延迟调度很敏感

**存储：**
- 需要维护 request state、历史、schema、cache 索引

**网络：**
- 单机时弱
- 多机 serving 时，调度器要把请求发往正确节点
- 若 prefill/decode 解耦，还要额外决定 KV 传输路径

TensorRT-LLM、vLLM、Dynamo 都把 request scheduling、continuous batching、KV cache 系统单独作为核心模块，而不是附属功能。([NVIDIA GitHub](https://nvidia.github.io/TensorRT-LLM/ "Welcome to TensorRT LLM’s Documentation! — TensorRT LLM"))
## 2) 上下文构造（context assembly）

### 输入
- 当前用户消息
- 系统指令
- 历史对话
- RAG 检索结果（如果有）
- 工具定义 / function signatures
- agent memory / scratchpad（如果是 agent）

### 中间结果
- 拼接后的 prompt
- role tags
- 工具列表的序列化表示
- 被截断或压缩后的历史
- 可能插入的检索文档

### 输出
- 统一文本或多模态 prompt

### 硬件影响
**计算：**
- 主要 CPU 侧字符串处理/模板填充
- 若要做 memory 检索、rerank，则可能引入额外 GPU/CPU 负载

**存储：**
- 这里决定后面 KV cache 大小
- prompt 越长，prefill FLOPs 越高，KV cache 越大

**网络：**
- 如果要从外部向量库/数据库取记忆，会增加网络 I/O
- agent 系统通常在这一步访问更多外部状态

这一阶段本身不重，但它决定后续 prefill 与 KV 的规模上限。

---
## 3) Tokenization + Embedding lookup
### 输入
- 拼接后的 prompt 文本

### 中间结果
- token IDs
- attention mask
- position ids
- 可能的 special tokens / tool tokens

### 输出
- token 序列
- 首层 embedding 输入向量

### 硬件影响
**计算：**
- tokenization 大多在 CPU
- embedding lookup 在 GPU 上通常不算主瓶颈
**存储：**
- 需要词表、embedding table
- 对大模型来说，相比后续权重和 KV，开销很小
**网络：**
- 基本无

---
## 4) Prefill：处理整段输入上下文

这是第一次真正重 GPU 的阶段。
### 输入
- 当前整段 token 序列
- 模型权重
- attention mask / positions

### 核心计算
对所有输入 token 做一次完整前向：
- embedding
- 每层 attention
- 每层 MLP
- 归一化、残差
- 输出 logits（通常只关心最后位置）
- 为每层每个 token 生成 K/V

### 中间结果
- 每层 hidden states
- 每层 attention 的 K/V
- 最终 logits
- 第一轮可采样 token 的概率分布

### 输出
- **KV cache**
- 首 token 的 logits
- 可能还包含 hidden state / pooled output（某些任务）

### 关键性质
- 所有输入 token 可并行处理
- 大矩阵乘较多
- 算术强度高
- 更偏 **compute-bound**

vLLM 和相关性能文档都把 prefill 视为 compute-bound，相对 decode 则是 memory-bound。([vLLM](https://docs.vllm.ai/en/stable/configuration/optimization/?utm_source=chatgpt.com "Optimization and Tuning - vLLM"))

### 硬件影响
计算：强**
- 吃 Tensor Core 吞吐
- 吃低精度矩阵乘（FP8/INT8/FP16/BF16）
- prompt 越长，FLOPs 越高
**存储：强**
- 需要读模型权重
- 需要写出整段 prompt 的 KV cache
- prompt 长度越大，KV 写入越大
**网络：中到强**
- 单卡弱
- 张量并行/流水并行时需跨卡同步
- 若采用 disaggregated prefill/decode，prefill 结束后还要把 KV 发给 decode 节点

TensorRT-LLM 文档、NVIDIA 关于 disaggregated serving 的文章都把 KV cache exchange 作为专门问题。([NVIDIA GitHub](https://nvidia.github.io/TensorRT-LLM/ "Welcome to TensorRT LLM’s Documentation! — TensorRT LLM")) 

## 5) cache 生成、管理与复用
这是现代推理系统最关键的“中间状态”。

### KV cache 是什么
对每层、每个 token，保存 attention 的 key/value 张量。  
后续 decode 生成新 token 时，不必重新计算历史 token 的 K/V，只需读取缓存并追加新的 K/V。

### 输入
- prefill 过程中每层产生的 K/V
### 中间结果
- 分块后的 KV blocks
- block table / page table
- 引用计数
- prefix cache 索引
- eviction metadata
### 输出
- 可复用的 KV cache
- prefix cache 命中基础

vLLM 的 Automatic Prefix Caching 文档明确说：如果新请求与旧请求有相同前缀，可以直接复用已有 KV cache，跳过共享前缀的计算。vLLM repo 也把 PagedAttention 和 KV 高效管理作为核心能力。([vLLM](https://docs.vllm.ai/en/latest/design/prefix_caching/?utm_source=chatgpt.com "Automatic Prefix Caching - vLLM"))：

### 硬件影响
计算: **
- 复用 KV 可直接减少 prefill 计算
**存储：极强**
- KV cache 常常是服务端最主要显存消费者之一
- 长上下文/高并发下，KV 比权重更先成为瓶颈
**网络：**
- 多机 disaggregated serving 时，需要跨节点传输 KV
- 多轮对话/agent 多次调用时，若策略不佳，反复 KV 传输会压满网络

关于这一点，2026 年的 PPD 论文直接指出：在多轮对话和 agentic 系统下，标准 Prefill-Decode 分离会反复重算与传输 KV，导致 turn 2+ TTFT 仍高，并可能饱和网络带宽。([arXiv](https://arxiv.org/html/2603.13358v1 "Not All Prefills Are Equal: PPD Disaggregation for Multi-turn LLM Serving"))

## 6) code 循环：逐 token 自回归生成
### 输入
- 当前最后一个 token
- 历史 KV cache
- 模型权重

### 单步内部过程

对第 t 个新 token：
1. 读取该 token embedding
2. 通过所有层前向
3. attention 读取**全部历史 KV cache**
4. 生成当前 token 对应的新 K/V
5. 追加到 KV cache
6. 产生新 logits
7. sampling 得到下一个 token

### 中间结果
- 当前步 hidden states
- 当前步新 K/V
- 当前步 logits
- 采样出来的新 token

### 输出
- 一个新 token
- 扩大的 KV cache

### 关键性质
- 每次只生成少量 token
- 步数多
- 每一步都要读大量权重与历史 KV
- 常见瓶颈是 **memory-bandwidth-bound**

vLLM 的 speculative decoding 文档明确说，它针对的是 **memory-bound workloads** 下的 inter-token latency 优化。([vLLM](https://docs.vllm.ai/en/latest/features/speculative_decoding/?utm_source=chatgpt.com "Speculative Decoding - vLLM"))：

### 硬件影响: 
计算: 中
- 每步 FLOPs 不算特别大
- GPU 往往不是“算不动”，而是“等数据”
**存储：极强**
- 高频读取权重
- 高频读取 KV cache
- 高频追加少量新 KV
- 因此非常吃 **HBM 带宽**
**网络：中到强**
- 张量并行时，每个 step 都有跨卡同步
- MoE 时还要做 expert routing 通信
- 多机部署下 decode 的细粒度通信对延迟很敏感

## 7) Sampling / 约束解码 / 工具选择

### 输入
- 当前 logits
- sampling 参数
- stop criteria
- JSON schema / grammar / function signatures
### 中间结果
- top-k / top-p 候选
- 温度缩放
- function/tool token 触发
- constrained decoder state
### 输出
- 一个 token 或一个结构化调用决策

### 硬件影响
**计算：**
- 通常不大
- 但在严格 JSON / grammar constrained decoding 下，CPU/GPU 都会有额外开销
**存储：**
- schema / grammar automaton 状态
**网络：**
- 若选择“调用工具”，下一步会引出外部网络 I/O

TensorRT-LLM 将 sampling 与 guided decoding 作为独立能力，说明这一步在高性能服务中也是专门优化对象。([NVIDIA GitHub](https://nvidia.github.io/TensorRT-LLM/ "Welcome to TensorRT LLM’s Documentation! — TensorRT LLM"))

## 8) tokenize / 输出后处理

### 输入
- 生成 token 流
### 中间结果
- UTF-8 文本
- JSON 结构
- 工具参数对象
- 流式 chunk
### 输出
- 用户可见文本
- 或工具调用 payload

### 硬件影响
**计算：**
- 主要 CPU
- 对小模型/短回答时，可能占端到端延迟的非小部分

**存储 / 网络：**
- 若流式返回，需要持续 socket/network 发送

# 二、如果是 Agentic 系统，推理链路如何改变

普通聊天：
> 一次上下文构造 → 一次 prefill → 多步 decode → 返回答案

> 上下文构造 → prefill → decode → 产出“工具调用” → 外部环境执行 → 工具结果回写 → 再次 prefill → 再次 decode → … → 直到结束

# 四、Agentic 对硬件的真实影响

## 1) 对 GPU 计算的影响

### 改变点
Agent 通常**不改变单次 Transformer 前向的计算图**。  
它改变的是：
- LLM 调用次数增多
- 平均上下文更长
- 工具结果不断塞回 prompt
- 多轮推理链更长

### 结果
- 总 FLOPs 上升
- 但未必线性，因为可通过 prefix/KV reuse 降低重复计算
- 推理型 agent 往往更偏长 decode 与多次小 prefill

### 典型现象
原本一次回答只需：
- 1 次 prefill
- 150 个 decode steps

agent 可能变成：
- 6 次 prefill
- 6 段 decode
- 3 次工具等待
- 2 次重试

于是端到端延迟里，GPU 计算占比反而可能下降，系统 orchestration 占比上升。
## 2) 对显存容量的影响
### 增加点
- 每一轮调用都要维护上下文
- 更长历史 + 工具输出 + planning scratchpad
- 更容易触发大 KV cache
### 结果
- agent 负载更容易受 **KV cache 容量** 限制
- 多 agent 并发时，显存不只装权重，还要装大量活跃会话 KV

### 特别重要的一点
多轮 agent 往往不是每轮都“全新问题”。  
如果系统不能有效复用已有 KV，就会频繁重算历史，浪费大量 prefill FLOPs 和显存带宽。PPD 论文正是针对多轮/agentic 场景指出这一点。([arXiv](https://arxiv.org/html/2603.13358v1 "Not All Prefills Are Equal: PPD Disaggregation for Multi-turn LLM Serving"))

## 3) 对内存带宽的影响  
agent 负载常使 decode 更突出，因为：
- 工具调用前后会多次“短生成”
- 每次都要读权重 + 历史 KV
- reasoning / planning 让输出链条变长

所以 agent 并不会让系统“更 compute-bound”，很多时候反而让**小段 decode 高频发生**，加剧 memory-bound 特征。

## 4) 对 CPU / RAM 的影响

这一点经常被低估。
Agent 额外需要 CPU 处理：
- 工具协议解析
- JSON schema 校验
- 浏览器/OS 自动化
- 检索与 rerank
- 任务图调度
- 日志与回放
- 权限与安全检查

所以 agent 系统通常比纯聊天系统更依赖：
- 更强 CPU 单核/多核
- 更大系统内存
- 更好的本地 SSD / 状态存储

## 5) 对网络的影响
Agent 对网络的影响常常比对 GPU 更大。
### 原因
它引入了三类网络：
#### A. 模型服务网络
- 客户端 ↔ 推理服务
- prefill ↔ decode 节点 KV 传输
- 多卡/多机同步

#### B. 工具/API 网络
- 浏览器访问网页
- 调外部 API
- SaaS / 数据库 / 向量库

#### C. 状态同步网络
- memory store
- event bus
- tracing / observability
- 队列系统

在 disaggregated serving 下，KV 传输已是问题；在 agent 多轮场景里，反复把新 turn 的内容送去 prefill，再把 KV 送回 decode，会进一步放大网络压力。([arXiv](https://arxiv.org/html/2603.13358v1 "Not All Prefills Are Equal: PPD Disaggregation for Multi-turn LLM Serving"))

# 五、agent 会改变“最优系统架构”

## 普通 chat

优化目标通常是：
- TTFT
- tokens/s
- GPU 利用率

## Agentic
还要加上：
- tool latency
- retry rate
- environment observation freshness
- planner efficiency
- KV reuse across turns
- external I/O overlap

因此最佳架构从“单模型高吞吐服务”变成“分层系统”：
1. **调度层**：任务分发、失败重试、预算控制
2. **推理层**：prefill/decode engine
3. **记忆层**：短期状态、长期记忆、prefix/KV 索引
4. **执行层**：browser、shell、API、workflow engine
5. **观测层**：trace、logs、cost、token accounting

OpenClaw 的“gateway + assistant + channels + skills”结构，本质上就体现了这种分层。([GitHub](https://github.com/openclaw/openclaw "GitHub - openclaw/openclaw: Your own personal AI assistant. Any OS. Any Platform. The lobster way.  · GitHub"))

# 六、硬件视角，agent 带来的几个关键变化
## 变化 1：GPU 不再是唯一瓶颈
纯 LLM 服务常主要看 GPU。  
Agent 系统里，瓶颈可能在：
- 浏览器自动化
- 外部 API 慢响应
- memory store
- JSON / schema 校验
- tool sandbox
所以端到端 latency 的贡献更分散。

## 变化 2：更需要 KV 复用与前缀缓存
多轮 agent 有大量重复上下文：
- system prompt
- tool spec
- memory header
- 历史计划模板

这使 prefix caching 的价值更高。vLLM 直接说明 APC 可以复用共享前缀的 KV，跳过那部分计算。([vLLM](https://docs.vllm.ai/en/latest/design/prefix_caching/?utm_source=chatgpt.com "Automatic Prefix Caching - vLLM"))

## 变化3: Prefill/decode 解耦  
因为 agent 的工作负载更混合：
- 有些轮次是长 observation → 重 prefill
- 有些轮次是短 decision → 重 decode latency
所以把 prefill 和 decode 分到不同 GPU 池，更容易优化。TensorRT-LLM 与 NVIDIA 的 disaggregated serving 方向就是为这种异构负载服务。([NVIDIA GitHub](https://nvidia.github.io/TensorRT-LLM/ "Welcome to TensorRT LLM’s Documentation! — TensorRT LLM"))

## 变化4: 存储多层而非仅显存  
Agent 需要：
- conversation state
- execution trace
- memory embeddings
- tool results cache
- checkpoints

这使 SSD / RAM / 外部存储的重要性上升。  
换句话说，agent 系统比纯聊天系统更像“分布式应用”，没那么像“单一 GPU kernel”。

## 1) 不改变的

- 还是 KV cache

- 工具总线

而不是“一个新的推理内核”。GitHub 和相关说明都强调 skills、channels、gateway、workflow shell 这类组件。([GitHub](https://github.com/openclaw/openclaw "GitHub - openclaw/openclaw: Your own personal AI assistant. Any OS. Any Platform. The lobster way.  · GitHub"))

# 十、最关键的结论

## 结论 1
**标准 LLM 推理的核心中间状态是 KV cache。**  
没有它，decode 成本会高得多；有了它，显存容量和带宽就成为核心约束。([vLLM](https://docs.vllm.ai/en/latest/design/prefix_caching/?utm_source=chatgpt.com "Automatic Prefix Caching - vLLM"))
## 结论2 
Prefill 与 decode 的硬件诉求不同。**
- prefill：更偏算力
- decode：更偏带宽
- 多轮/agent：更偏 KV 复用与网络协同([vLLM](https://docs.vllm.ai/en/stable/configuration/optimization/?utm_source=chatgpt.com "Optimization and Tuning - vLLM"))
## 结论3
OpennClaw 这类 agent 框架不主要改变单步 Transformer 计算，而是把单次推理变成多轮控制循环。它增加的是：
- 工具调用
- 状态持久化
- 外部观测
- 多次推理回合
- 更强的 CPU/存储/网络需求([GitHub](https://github.com/openclaw/openclaw "GitHub - openclaw/openclaw: Your own personal AI assistant. Any OS. Any Platform. The lobster way.  · GitHub"))
## 结论4
- Agentic 系统的最优基础设施不是“最大 GPU”，而是“推理层 + 存储层 + 工具层 + 调度层”的整体设计。

# 一、先给总框架：一次“完整推理”到底是什么

标准大模型在线推理，不是单个“forward”这么简单，而是一个多层流水线：
**请求进入 → 上下文构造 → 分词 → Prefill → 生成并保存 KV cache → Decode 循环 → 采样/约束解码 → detokenize/输出**。  

如果是 agentic 系统，还会继续变成：
**输出工具调用 → 外部环境执行 → 工具结果回注入上下文 → 再次 Prefill → 再次 Decode → …直到任务结束**。

现代 serving 框架公开文档都已把这类系统问题作为核心能力来处理，而不只是“加载模型然后生成文本”：例如 vLLM 把 paged attention、prefix caching、KV 管理作为核心设计；TensorRT-LLM 把 in-flight batching、paged attention、KV cache reuse、disaggregated serving、KV cache exchange 列为关键特性。([vLLM](https://docs.vllm.ai/en/latest/design/paged_attention/?utm_source=chatgpt.com "Paged Attention - vLLM"))

# 二、模型内部的完整工作过程

## 1. 输入构造阶段

### 输入
这一阶段的原始输入通常包括：
- system prompt
- 用户消息
- 历史对话
- 工具定义 / function schema
- 结构化输出约束（JSON schema / grammar）
- 检索结果、记忆结果、外部文档摘要（如果有）

### 中间结果
系统会把这些内容拼成最终 prompt。  
这一步的中间结果不是神经网络张量，而是：
- 拼接后的文本序列
- 被裁剪后的历史
- 被序列化的工具说明
- memory / RAG 片段
- 一些特殊 token 标记

### 输出
最终输出是一个待分词的 prompt 文本或多模态输入序列。

### 对硬件的影响
这一步主要发生在 CPU 与内存中。  GPU 几乎不重，但它决定后面两件大事：
1. **Prefill 计算量**：prompt 越长，Prefill FLOPs 越大。
2. **KV cache 大小**：上下文越长，后续需要保存和读取的 KV 越大。
所以它虽然不重算力，却在决定后续 GPU 成本的上限。

## 2. Tokenization 与输入张量准备
### 输入
拼好的文本 prompt。
### 中间结果
文本被 tokenizer 转成 token IDs，同时生成：
- `input_ids`
- `attention_mask`
- `position_ids`
- 可能的 tool / special tokens
- 有些模型还会生成 rope 位置编码相关参数
### 输出
这些 token IDs 会被送入 embedding 层，变成第一层 hidden states。
### 对硬件的影响
tokenization 通常主要是 CPU 工作。  
embedding lookup 会在 GPU 上进行，但相较后面的 attention/MLP，通常不是主瓶颈。  
这一阶段对网络几乎没有影响。

## 3. Prefill：把整段上下文“读完”
这是很多解释里最容易被略过、但其实最关键的一步。
### 输入
- 整段 prompt 的 token IDs
- 模型全部权重
- position / mask 信息

### 计算过程
对长度为 `L` 的输入序列，模型会对这 `L` 个 token 一次性做完整前向。  
以第 `ℓ` 层为例，通常会发生：
1. 输入 hidden state：形状可抽象为 `[batch, seq_len, hidden_dim]`
2. 经过线性投影得到 `Q, K, V`
3. 做 attention：
    - `Q` 与同序列内所有历史位置的 `K` 交互
    - 加权求和得到 attention output
4. 经过输出投影
5. 残差连接 + 归一化
6. 进入 MLP / SwiGLU / FFN
7. 再残差连接 + 归一化
8. 输出下一层 hidden state

这在所有层重复后，最后一层 hidden state 会经过 lm head 得到 logits。

### 关键中间结果
Prefill 结束时，产生三个关键中间结果：
#### A. 最后一位置的 logits
这是用来生成首个输出 token 的概率分布。
#### B. 每层每个 token 的 K/V
这是最重要的中间状态。  
这些 `K` 和 `V` 不会丢弃，而是保存为 **KV cache**。
#### C. 一系列中间 hidden states
通常不长久保存，除非特定优化或特殊任务需要。
### 输出
- 首 token 的 logits
- 全层 KV cache
- 当前请求的状态元数据

### 为什么 Prefill 更偏算力
因为整段输入可以并行处理，attention 与 MLP 都能以较大的矩阵乘形式运行，Tensor Core 利用率高。  
vLLM 的优化文档直接把 prefill 和 decode 视为不同负载；实践上，Prefill 通常更接近 **compute-bound**。([vLLM](https://docs.vllm.ai/en/stable/configuration/optimization/?utm_source=chatgpt.com "Optimization and Tuning - vLLM"))

### 对硬件的影响
#### 计算
- 非常吃 GPU 矩阵乘吞吐
- 吃 Tensor Core
- 吃低精度推理能力（FP16/BF16/FP8/INT8 等）
#### 存储
- 需要读全模型权重
- 需要把整段 prompt 对应的 KV cache 写入显存
- prompt 越长，KV 写入越大

#### 网络
- 单卡时几乎无外部网络问题
- 多卡张量并行时需要跨卡同步
- 若是 disaggregated serving，还可能需要把 Prefill 产生的 KV 发送给 Decode 节点。TensorRT-LLM 文档把 KV cache exchange 专门单列出来，说明这已是独立系统问题。([NVIDIA GitHub](https://nvidia.github.io/TensorRT-LLM/1.2.0rc6/features/disagg-serving.html?utm_source=chatgpt.com "Disaggregated Serving — TensorRT LLM"))

## 4. KV cache：推理中最关键的长期中间状态
很多人把“输出 token”当成推理的产物，但工程上更关键的产物其实是 KV cache。
### 它是什么
在每一层 attention 中，模型会为每个 token 生成对应的 `K` 和 `V` 张量。  
在 decode 时，不需要重新计算历史 token 的 `K/V`，只需把新 token 的 `Q` 与历史缓存的 `K/V` 交互即可。  
这就是自回归推理能成立的关键。
### 输入
Prefill 过程中每层生成的 `K` 和 `V`。
### 中间结果
在现代系统里，KV cache 通常不是简单的一整块连续数组，而是分页/分块存储：
- page/block table
- block allocator
- eviction metadata
- prefix reuse 索引
- 对 sliding window / hybrid attention 的单独管理
vLLM 的 paged attention 和 hybrid KV cache manager 文档都明确说明了这类设计。([vLLM](https://docs.vllm.ai/en/latest/design/paged_attention/?utm_source=chatgpt.com "Paged Attention - vLLM"))
### 输出
- 当前会话可持续追加的 KV cache
- 可能被 prefix caching 复用的缓存块
### 为什么它重要
NVIDIA 的推理优化文章直接指出：LLM 推理显存的两个主要部分就是 **模型权重** 和 **KV cache**。([NVIDIA Developer](https://developer.nvidia.com/blog/mastering-llm-techniques-inference-optimization/?utm_source=chatgpt.com "Mastering LLM Techniques: Inference Optimization"))
### 对硬件的影响
#### 计算
KV cache 本身不是额外计算负担，但它减少了重复计算。
#### 存储
这是大头之一：
- 短上下文时，权重可能占主导
- 长上下文 / 高并发时，KV cache 很容易成为主导显存开销
vLLM 的调优文档也直接说明：提高 `gpu_memory_utilization` 可以给 KV cache 更多空间；降低并发序列数可降低 KV 占用。([vLLM](https://docs.vllm.ai/en/stable/configuration/optimization/?utm_source=chatgpt.com "Optimization and Tuning - vLLM"))
#### 网络
如果 Prefill 与 Decode 分离，KV 需要跨节点传输。  
TensorRT-LLM 已把这件事作为独立模块来做。([NVIDIA GitHub](https://nvidia.github.io/TensorRT-LLM/1.2.0rc6/features/disagg-serving.html?utm_source=chatgpt.com "Disaggregated Serving — TensorRT LLM"))

## 5. Decode：逐 token 自回归生成
这是在线服务里更常见的持续瓶颈。
### 输入
- 最新 token
- 历史 KV cache
- 模型权重
- 采样参数
### 每一步内部发生什么
对每一个新 token，模型都会做一轮小型前向：
1. 读取这个 token 的 embedding
2. 通过所有层
3. 在每一层里，用当前 token 的 `Q` 去和**全部历史 token 的 K/V cache** 做 attention
4. 生成该 token 的新 `K/V`
5. 追加到 KV cache
6. 输出当前步 logits
7. 经过 sampling 得到下一个 token
### 中间结果
- 当前步的 hidden state
- 当前步新生成的 K/V
- 当前步 logits
- 当前采样出的 token
### 输出
- 一个或少数几个新 token
- 扩展后的 KV cache

### 为什么 Decode 更偏带宽
每一步的矩阵计算量不算极大，但要反复：
- 读取权重
- 读取全部历史 KV
- 追加新 KV

这使它常成为 **memory-bandwidth-bound**。  
vLLM 关于 speculative decoding 的文档也明确把它描述为针对 memory-bound LLM inference 的优化。([vLLM](https://docs.vllm.ai/en/stable/configuration/optimization/?utm_source=chatgpt.com "Optimization and Tuning - vLLM"))

### 对硬件的影响
#### 计算
- 单步 FLOPs 小于 Prefill
- 但总步数多
- reasoning 模型会把这部分显著放大
#### 存储
- 高频读取 HBM 中的权重和 KV
- 极度依赖 HBM 带宽
- 长上下文会显著放大 KV 读取量
#### 网络
- 张量并行时，每一步可能都要通信
- 多机时，细粒度同步非常伤延迟
- MoE 时还会叠加专家路由通信

## 6. Sampling、约束解码与结构化输出

### 输入
- 当前 logits
- top-k、top-p、temperature、repetition penalty
- JSON schema / grammar / tool definitions
### 中间结果
- 候选 token 集合
- 过滤后的概率分布
- grammar automaton 状态
- 是否触发工具调用
### 输出
- 下一个 token
- 或结构化函数调用意图

### 对硬件的影响

#### 计算
通常不如 attention/MLP 重，但当使用严格 JSON / grammar constrained decoding 时，这部分的 CPU/GPU 开销会上升。
#### 存储
主要是小规模状态机和 schema。
#### 网络
如果输出的是工具调用，那么下一步开始引入外部网络 I/O。

## 7. 输出后处理
### 输入
生成的 token 序列。
### 中间结果
- detokenized 文本
- JSON 对象
- 工具参数
- 流式分块响应
### 输出
用户看到的最终文本，或者发送给工具执行器的调用指令。
### 对硬件的影响
主要是 CPU 与网络发送，GPU 影响很小。

# 三、系统层：现代 serving 还做了哪些隐藏步骤

前面讲的是“单请求的神经网络链路”。但 SOTA 系统还有一整层 serving 逻辑。

## 1. Continuous / In-flight batching
不是把一批请求一起做完才下一批，而是动态把新请求插入 GPU 执行窗口。  
TensorRT-LLM 文档明确把 in-flight batching 作为关键能力。([NVIDIA GitHub](https://nvidia.github.io/TensorRT-LLM/overview.html?utm_source=chatgpt.com "Overview — TensorRT LLM"))
### 硬件意义
- 提高 GPU 利用率
- 让 Prefill 与 Decode 混合并发
- 但也增加 KV 管理和调度复杂度

## 2. Paged Attention
KV cache 不是按请求一次性整块连续分配，而是分页/分块管理。  这是 vLLM 的核心设计之一。([vLLM](https://docs.vllm.ai/en/latest/design/paged_attention/?utm_source=chatgpt.com "Paged Attention - vLLM"))
### 硬件意义
- 减少显存碎片
- 支持高并发
- 便于 prefix reuse
- 便于 eviction

## 3. Prefix Caching
如果多个请求共享相同前缀，可以直接复用历史 KV。  
vLLM 文档明确说明 APC 会缓存已有 query 的 KV cache，让新 query 跳过共享前缀的计算。([vLLM](https://docs.vllm.ai/en/latest/design/prefix_caching/?utm_source=chatgpt.com "Automatic Prefix Caching - vLLM"))
### 硬件意义
- 降低 Prefill FLOPs
- 降低首 token 延迟
- 降低重复显存写入
- 对 system prompt 很长、工具 schema 很长、agent memory header 很长的场景特别有效

## 4. Prefill / Decode 解耦
TensorRT-LLM 文档专门描述了 disaggregated serving 和 KV cache exchange。([NVIDIA GitHub](https://nvidia.github.io/TensorRT-LLM/?utm_source=chatgpt.com "Welcome to TensorRT LLM's Documentation!"))
### 硬件意义
因为二者资源画像不同：
- Prefill 更像算力密集
- Decode 更像带宽/显存密集
把它们拆到不同 GPU 池，可以分别优化。

# 四、并行策略如何改变硬件需求

## 1. 单卡
最简单。  瓶颈主要是：
- HBM 容量
- HBM 带宽
- Tensor Core 吞吐
## 2. 张量并行（Tensor Parallel）
同一层权重切到多卡。  每层都需要做跨卡汇总。
### 影响
- 能装更大模型
- 但每步通信更多
- Decode 尤其受通信延迟影响
## 3. 流水并行（Pipeline Parallel）
不同层放不同卡。  激活在 stage 间流动。
### 影响
- 能装更深/更大模型
- 增加流水线气泡和跨卡传输

## 4. 专家并行（Expert Parallel / MoE）

只激活少数专家，但 token 需要路由到不同专家设备。
### 影响
- 每 token 有效计算量可下降
- 但会引入显著 all-to-all 通信
- 系统从“算力/带宽问题”进一步变成“互连拓扑问题”

# 五、为什么有些模型结构会改变推理瓶颈
## 1. MQA / GQA
NVIDIA 的相关技术文章明确指出：Multi-Query Attention 可以减少 KV cache 大小，从而降低增量解码的带宽需求。([NVIDIA Developer](https://developer.nvidia.com/blog/mastering-llm-techniques-training/?utm_source=chatgpt.com "Mastering LLM Techniques: Training | NVIDIA Technical Blog"))
### 硬件意义
- 减少 KV 显存占用
- 减少 decode 访存压力
- 更有利于长上下文和高并发
## 2. 量化
量化不仅能减权重容量，还能减 KV cache 容量。  
NVIDIA 2025 年关于 NVFP4 KV cache 的文章就直接把减少权重、激活和 KV 精度与吞吐、延迟、可实现上下文长度改善联系起来。([NVIDIA Developer Forums](https://forums.developer.nvidia.com/t/optimizing-inference-for-long-context-and-large-batch-sizes-with-nvfp4-kv-cache/353975?utm_source=chatgpt.com "Optimizing Inference for Long Context and Large Batch ..."))
### 硬件意义
- 降显存容量
- 降 HBM 读写压力
- 可能提高吞吐
- 但取决于硬件是否原生支持该精度
# 六、Agentic 系统到底改了什么
## 1. 没改什么
通常**没有改写基础 Transformer 的单步数学过程**。  仍然是：
- embedding
- attention
- MLP
- KV cache
- prefill
- decode
所以 OpenClaw 这类东西不是“新的推理内核”。
## 2. 改了什么
它把原本的一次性生成，变成外层控制循环：
**计划 → 生成一个动作 → 调工具 → 获取环境反馈 → 再计划 → 再生成 → …**
OpenClaw 的公开说明里，重点是：
- 多消息渠道
- control plane / gateway
- skills / tools
- memory / assistant 层
- 本地设备运行与交互能力  
    这些都说明它是 agent harness / assistant shell，而非基础模型本身。([GitHub](https://github.com/openclaw/openclaw?utm_source=chatgpt.com "OpenClaw — Personal AI Assistant"))

# 七、Agentic 推理的完整工作过程

下面给一条完整 agent 回环。
## 第 0 步：任务输入
用户不是给一个单问题，而是给一个目标。  
例如：“下载财报，提取电话会要点，写成摘要并发到 Slack。”
## 第 1 步：任务规划
LLM 先根据工具列表和任务目标生成一个计划。
### 输入
- 任务目标
- 工具定义
- 历史状态
- 权限信息
- 记忆
### 输出
- 计划草案
- 下一步动作选择
### 硬件影响
多一次 LLM 调用；通常上下文更长，因为要带工具规范。
## 第 2 步：工具调用生成
模型输出 function/tool call。
### 输出
- API 参数
- 浏览器动作
- shell 命令
- 文件操作请求
### 硬件影响
LLM 本身计算不特殊，但会引出外部 I/O。
## 第 3 步：环境执行
真正的 browser / shell / API / DB 执行器去做动作。
### 输出
- HTML/DOM
- 文件内容
- API JSON
- 错误码
- 屏幕观察结果
### 硬件影响
此时瓶颈经常不在 GPU，而在：
- 浏览器自动化
- 网络 RTT
- 文件系统
- 外部服务延迟

## 第 4 步：结果回注入上下文
工具结果被压缩、摘要或原样回注入 prompt。
### 后果
- 上下文变长
- 新一轮 Prefill 变重
- KV cache 继续膨胀

## 第 5 步：再次推理
模型基于新观察继续做：
- 判断是否成功
- 是否重试
- 是否换工具
- 是否输出最终答案
然后重复循环。
# 八、Agentic 对硬件的具体影响

## 1. 对 GPU 计算
### 变化
单次前向的计算图不变，但**总调用次数明显增加**。  
原来可能只需一次 Prefill + 一段 Decode；现在可能变成 5 次 Prefill + 5 段 Decode。
### 结果
- 总 FLOPs 上升
- 首 token 延迟更频繁出现
- 长 system prompt / tool schema 会反复进入 Prefill，除非 prefix cache 命中
## 2. 对显存容量
Agent 往往携带更多上下文：
- 工具列表
- 计划文本
- 观察结果
- 记忆摘要
- 中间轨迹
这会显著增加 KV cache 占用。  
因此 agent 系统更容易先撞到 **KV 容量瓶颈**，而不是仅仅撞到权重容量。
## 3. 对显存带宽
多轮小步生成会让 decode 更频繁发生。  
如果 agent 需要大量“想一步、做一步、再想一步”，那就是频繁短 decode，对带宽与调度都不友好。
## 4. 对 CPU / RAM
这是前一版漏得较多的地方。Agent 系统额外需要：
- 工具参数解析
- JSON/schema 校验
- 浏览器自动化
- 任务图调度
- 重试与异常处理
- 状态持久化
- 日志与可观测性

所以 agent 系统比纯聊天系统更依赖：
- CPU 核数与单核性能
- 大内存
- SSD / 本地存储
- 外部数据库与向量库
## 5. 对网络
Agent 系统新增三层网络压力：
### A. 推理服务内部网络
- 多机推理通信
- Prefill/Decode 间 KV 传输
### B. 工具网络
- Web/API 请求
- 数据库访问
- 第三方 SaaS
### C. 状态同步网络
- 记忆层
- 事件总线
- tracing / metrics
- 
所以 agent 系统的端到端瓶颈经常从“GPU 不够快”转成“GPU + 网络 + 工具 I/O + orchestration 共同限制”。
# 九、Agentic 为什么会放大 KV 复用的重要性

这是最容易被忽略但最关键的一点。Agent 的多轮调用通常具有大段共享前缀，例如：
- 固定 system prompt
- 固定工具定义
- 固定安全规则
- 固定 memory header
- 大量相同历史

因此，如果没有 prefix caching / KV reuse，每一轮都要重新 Prefill 这些共享部分。  
vLLM 的 APC 正是为这种共享前缀场景设计的。([vLLM](https://docs.vllm.ai/en/latest/design/prefix_caching/?utm_source=chatgpt.com "Automatic Prefix Caching - vLLM"))

所以对 agent 系统而言，prefix cache 的价值通常**高于**普通单轮聊天。

# 十、把标准 chat 与 OpenClaw 类 agent 做成对照

## 1. 普通 chat
### 过程
一次上下文构造 → 一次 Prefill → 一次 Decode 流

## 2. OpenClaw / agentic

# 十一、最终可以怎样概括
## 标准 LLM 推理
可以概括为：
**把输入上下文转成 token，做一次高并行 Prefill，生成 KV cache；再做多轮低算术强度、强带宽依赖的 Decode；期间不断读取权重和 KV，并逐步扩展 KV。**

- **Prefill** 更偏 GPU 计算吞吐

-

# Gemini.Prompt: 大模型推理主要步骤和对硬件架构的需求 

在目前的大模型（LLM）推理架构中，推理过程主要被划分为两个截然不同的阶段：**Prefill（预填充）** 和 **Decoding（解码）**。这两个阶段在计算模式和硬件瓶颈上存在显著差异。

## 1. Prefill 阶段 (预填充 / Prompt 处理)

这是处理用户输入（Prompt）的第一个步骤。模型一次性并行读入所有输入 Token，计算它们的隐藏状态并生成 **KV Cache**（键值缓存）。

- **计算任务：** 大规模矩阵乘法（GEMM）。因为所有输入 Token 都是已知的，可以进行高度并行化处理。
    
- **硬件需求特点：**
    
    - **计算密型 (Compute-Bound)：** 该阶段能充分榨干 GPU 的算力（TFLOPS）。
        
    - **高算力需求：** 硬件的峰值计算性能直接决定了“首字延迟”（TTFT, Time to First Token）。
        
    - **内存带宽要求：** 虽然也需要带宽，但由于计算密度（Arithmetic Intensity）高，算力通常是主要瓶颈。
        

## 2. Decoding 阶段 (解码 / 逐字生成)

在生成首个 Token 后，模型进入自回归模式，根据已有的上下文逐个预测下一个 Token。

- **计算任务：** 向量-矩阵乘法（GEMV）。每一步只处理一个新 Token，但必须读取之前所有 Token 的 KV Cache 以及全部模型权重。
    
- **硬件需求特点：**
    
    - **访存密集型 (Memory-Bound)：** 这是目前大模型推理最大的痛点。计算单元（Tensor Cores）大部分时间在等待数据从显存（VRAM）传输进来。
        
    - **高存储带宽需求：** 显存带宽（GB/s）直接决定了生成速度（Tokens per second）。这也是为什么配备 HBM（高带宽内存）的 GPU（如 H100, A100）比普通显卡快得多的核心原因。
        
    - **低算力利用率：** 在此阶段，硬件的峰值算力往往被浪费。
        

---

## 硬件架构需求对比表

|**维度**|**Prefill 阶段 (首字)**|**Decoding 阶段 (后续字)**|
|---|---|---|
|**主要瓶颈**|**算力 (Compute)**|**显存带宽 (Memory Bandwidth)**|
|**计算模式**|计算密集、高度并行|访存密集、串行依赖|
|**存储容量需求**|主要是模型权重 + 激活值|**KV Cache** 随上下文长度线性增长|
|**关键硬件指标**|TFLOPS (FP16/INT8)|HBM 带宽 (GB/s)|
|**优化方向**|算子融合 (Kernel Fusion)|量化 (Quantization)、连续批处理 (Continuous Batching)|

---

## 关键硬件参数的影响分析

1. **存储容量 (Memory Capacity):**
    
    - **模型权重：** 例如 Llama-3-70B (FP16) 需要约 140GB 显存才能跑起来。
        
    - **KV Cache：** 这是长文本推理的杀手。随着上下文增加，KV Cache 会迅速吞噬显存。对于 128K 上下文，KV Cache 的大小可能接近甚至超过模型权重本身。
        
2. **存储带宽 (Memory Bandwidth):**
    
    - 在 Decoding 阶段，即使你有再强的算力，如果带宽不足（如普通 DDR5 内存），Token 生成速度可能只有每秒 1-2 个，远低于人类阅读速度。
        
3. **计算量 (FLOPS):**
    
    - 主要影响高并发场景和 Prefill 速度。在处理长 Prompt 或多用户同时请求时，高算力能显著缩短响应启动时间。

# Gemini.Prompt: Prefill阶段之后, KV Cache里面保存的是什么内容? 

在 Prefill 阶段完成后，**KV Cache**（键值缓存）中保存的是输入序列中所有 Token 的 **Key（键）向量**和 **Value（值）向量**。

为了理解其本质，我们需要从 Transformer 的自注意力机制（Self-Attention）物理意义出发：

## 1. KV Cache 到底存了什么？

在 Transformer 层中，每个 Token 会经过线性变换生成三个向量：$Q$ (Query), $K$ (Key), $V$ (Value)。

- **Key (K) 向量：** 代表这个 Token 的“特征索引”或“身份标签”。它告诉后续生成的 Token：“我在这里，我具有这些语义特征”。
    
- **Value (V) 向量：** 代表这个 Token 包含的“实际信息内容”。
    

**在 Prefill 结束后：**

GPU 显存中会开辟一块空间，按层（Layer）和按头（Head）存储所有已处理 Token 的 $K$ 和 $V$。

- **数学形式：** 对于输入长度为 $L$ 的序列，KV Cache 存储的是两个矩阵 $K_{prefix} \in \mathbb{R}^{L \times d}$ 和 $V_{prefix} \in \mathbb{R}^{L \times d}$（其中 $d$ 是每个头的维度）。
    

---

## 2. 为什么要存这些内容？（逻辑必然性）

在随后的 **Decoding（解码）** 阶段，模型每产生一个新 Token（假设为 $T_{next}$），它都需要与之前所有的 Token 计算注意力：

1. **计算注意力分数：** 用新 Token 的 $Q_{next}$ 去乘以 KV Cache 中所有的 $K_{prefix}$。
    
2. **加权信息聚合：** 根据上一步的分数，对 KV Cache 中所有的 $V_{prefix}$ 进行加权求和。
    

**核心痛点：** 如果不存 KV Cache，每生成一个新 Token，模型都要重新计算一遍前面所有 Token 的 $K$ 和 $V$。这会导致计算量随序列长度呈**平方级 ($O(L^2)$)** 增长。通过存储 KV Cache，我们将重复计算转化为了**空间换时间**，使解码阶段的计算量降为 $O(L)$。

---

## 3. KV Cache 的数据特点与硬件压力

|**特点**|**详细描述**|**硬件影响**|
|---|---|---|
|**动态增长**|每生成一个 Token，KV Cache 就会增加一列。|需要复杂的显存管理（如 PagedAttention）。|
|**规模巨大**|对于 Llama-3 70B，128K 上下文的 KV Cache 可能超过 100GB。|**容量瓶颈**：显存不够会限制上下文长度。|
|**访存频繁**|每步解码都要读取全部 KV Cache。|**带宽瓶颈**：HBM 带宽决定了生成的 Token 速度。|

## 4. 存储内容的精简趋势

在 Rubin/Vera Rubin 架构中，为了减轻 KV Cache 的压力，采用了以下技术来改变存储内容：

- **GQA (Grouped Query Attention):** 多个 Query 共享一组 KV，使 KV Cache 体积缩小 4-8 倍。
    
- **量化 (Quantization):** 将 KV Cache 从 FP16 压缩为 **INT4** 或 **FP8**，直接减少一半以上的显存占用。

# Gemini.Prompt: 多个128K上下文的用户并发使用时, 每个用户的输入Query不同, KV Cache怎么共享复用? 

针对多个用户并发、输入 Query 不同的场景，**KV Cache 在本质上是无法“完全复用”的**，因为每个用户的 Prompt（上下文）具有独特性，生成的 $K$ 和 $V$ 向量自然不同。

但在工业级推理引擎（如 vLLM、TensorRT-LLM）和最新的 NVIDIA 架构中，通过以下几种高级技术实现了**部分复用**和**显存碎片化管理**，从而支撑高并发：

---

## 1. 核心技术：PagedAttention（显存的分页管理）

这是解决并发问题的基础。传统方式下，KV Cache 必须申请连续的显存空间，这会导致巨大的显存碎片（External Fragmentation）。

- **原理**：借鉴操作系统的虚拟内存思想，将 KV Cache 划分为固定大小的“块”（Blocks）。
    
- **效果**：不同用户的 KV Cache 可以物理上散落在显存各处，但在逻辑上是连续的。这允许系统根据用户请求**动态按需分配**，极大提升了并发量。
    

---

## 2. 公共前缀复用 (Prefix Caching)

虽然每个用户的 Query 不同，但在很多实际场景中，它们拥有**相同的公共前缀**。

- **场景示例**：
    
    - **系统提示词 (System Prompt)**：“你是一个资深的金融分析师，请根据以下报告回答问题...”
        
    - **长文档问答**：多个用户同时针对同一份 100K 字的财报进行不同维度的提问。
        
- **复用机制**：
    
    - 推理引擎会计算公共前缀的 Hash 值。
        
    - 如果多个用户的请求包含相同的 Hash 前缀，显存中只保留**一份**该前缀的 KV Cache。
        
    - 不同用户仅需为自己独特的 Query 部分申请额外的 KV Cache 空间。
        
- **硬件价值**：在 128K 这种长上下文中，如果公共前缀占 100K，复用能直接节省 90% 以上的显存。
    

---

## 3. 多租户下的 KV Cache 调度策略

当并发用户极多，显存依然不足时，系统会采用以下调度手段：

|**技术名称**|**实现方式**|**作用**|
|---|---|---|
|**Continuous Batching**|允许不同进度的请求（有的在 Prefill，有的在 Decoding）在同一个 Batch 中运行。|提高 GPU 算力利用率，减少排队延迟。|
|**Swapping (换入换出)**|当 GPU 显存满载时，将不活跃用户的 KV Cache 暂时移动到 **CPU 内存**或 **NVMe SSD**。|保证系统不崩溃，支持超大规模并发。|
|**Copy-on-Write**|当一个序列分叉时（如 Beam Search），先共享 KV Cache，只有当数据发生改变时才复制。|减少搜索算法下的显存占用。|

---

## 4. Rubin/Vera Rubin 架构下的物理增强：BlueField-4 STX

在最新的 Vera Rubin 架构白皮书中，NVIDIA 引入了硬件级的 KV Cache 管理器：

- **解耦存储**：KV Cache 不再仅仅堆存在 GPU 昂贵的 HBM 中，而是通过 **BlueField-4 STX (Storage/Network Offload)** 实时卸载到专用的 **CMX (Cache Management eXpansion)** 模块。
    
- **跨节点复用**：如果用户 A 在节点 1 提问，用户 B 在节点 2 提问同一个文档，STX 能够识别并直接通过高带宽网络调度已有的 KV 块，实现**集群级别的 Cache 共享**。
    

---

## 总结：如何实现“共享”？

1. **逻辑相同部分（System Prompt/共享文档）**：通过 **Prefix Caching** 实现 1:N 的物理共享。
    
2. **逻辑不同部分（User Query）**：无法共享，但通过 **PagedAttention** 实现显存的极致利用。
    
3. **计算资源**：通过 **Continuous Batching** 让所有用户共享 GPU 的计算流水线。

# Gemini.Prompt: agent涉及到非常多的工具调用, KV Cache有什么变化?
在 **Agent（智能体）** 架构下，频繁的工具调用（Tool Use / Function Calling）对 KV Cache 的管理带来了极具挑战性的变化。传统的线性对话模式变成了**非线性的、反复迭代的“回声”模式**。

以下是 Agent 场景下 KV Cache 的四大核心变化与挑战：

## 1. 螺旋式增长的上下文 (Spiral Context Growth)

普通的对话是“一问一答”，而 Agent 的工作流通常是：`思考 -> 调用工具 -> 获取结果 -> 重新思考`。

- **变化**：每一次工具返回的结果（如搜索结果、数据库查询、代码运行输出）都会被追加到 Prompt 中。
    
- **后果**：KV Cache 的需求不再是平滑增长，而是**阶跃式爆发**。如果一个 Agent 调用了 5 次工具，每次返回 2K Token 的数据，那么在最终推理前，KV Cache 已经额外吞噬了 10K Token 的空间。
    

## 2. 高频的“前缀重用” (High-Frequency Prefix Reuse)

Agent 通常携带复杂的 **System Prompt**（包含角色设定、工具列表说明、JSON 输出格式规范）。

- **变化**：在多轮工具调用中，这一部分长达数千 Token 的指令是**完全固定**的。
    
- **优化需求**：必须使用 **Prefix Caching**。如果没有缓存，Agent 每次“思考”都要重新 Prefill 遍工具定义，这会极大地增加首字延迟（TTFT）并浪费算力。
    

## 3. “分叉”与“回溯” (Context Branching & Backtracking)

复杂的 Agent（如使用 Tree of Thoughts 算法）会尝试多种执行路径，如果路径 A 失败，它会回溯到路径 B。

- **变化**：KV Cache 不再是一条直线，而是一棵**树（Tree Structure）**。
    
- **技术应对**：推理引擎需要支持 **Copy-on-Write (CoW)**。当 Agent 决定从第 500 个 Token 处尝试另一种工具调用时，系统只需指向已有的前 500 个 Token 的 KV 块，仅为新的分叉路径申请新空间。
    

## 4. 频繁的 Prefill 与 Decoding 切换

Agent 的本质是“小步快跑”：先做一小段 Decoding（生成工具调用指令），停下来等待外部结果，再进行一段大容量的 Prefill（读入工具返回结果）。

- **硬件压力**：这导致 GPU 在 **Compute-Bound (Prefill)** 和 **Memory-Bound (Decoding)** 之间频繁切换。
    
- **架构影响**：在 **Vera Rubin 架构** 中，这正是 **CPX** 和 **LPX** 协同工作的典型场景。CPX 负责快速消化工具返回的海量数据（Prefill），并将生成的 KV 块同步给 LPX 进行下一步动作决策（Decoding）。
    

---

## Agent 场景下 KV Cache 管理对比

|**维度**|**普通 Chat 模式**|**Agent 模式**|
|---|---|---|
|**增长曲线**|线性平稳|阶跃式、爆发式|
|**逻辑结构**|单线队列|树状结构 (Branching)|
|**System Prompt 占比**|较低|极高 (包含大量 API 定义)|
|**缓存敏感度**|中等|极高 (由于频繁循环，缓存失效会导致性能雪崩)|
|**典型瓶颈**|显存容量|显存带宽 + 调度开销|

## 总结

在 Agent 设计中，KV Cache 已经从单纯的“加速工具”变成了 **Agent 的“短时记忆载体”**。为了支撑复杂的 Agent 逻辑，现代参考架构越来越依赖 **PagedAttention** 来处理碎片化显存，以及 **FlashAttention-3** 等算子来加速频繁发生的 Prefill 过程。
