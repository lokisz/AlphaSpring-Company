# Prompt: SOTA大语言模型的训练分为哪几个主要阶段, 主要工作内容是? 各个阶段有什么输入输出?

先校正前提：
1. **不存在单一、固定的“SOTA大语言模型训练标准流程”**。不同机构会在后训练阶段采用不同路线，例如 RLHF、RLAIF、Constitutional AI、DPO 类偏好优化、高算力 RL、工具使用训练、多模态扩展等。
2. 但从公开资料看，当前前沿模型通常都可以抽象成两大层、五个常见阶段：  
    **数据准备 → 预训练 → 指令/行为监督微调 → 偏好对齐/强化学习 → 安全与专项能力后处理**。  
    OpenAI 近年的公开表述也明确把流程概括为 **pre-training + post-training** 两大部分；GPT-4.5 提到扩大了 pre-training 和 post-training，gpt-oss 也明确提到其 post-training 包含 **SFT + 高算力 RL**。([OpenAI](https://openai.com/index/introducing-gpt-4-5/?utm_source=chatgpt.com "Introducing GPT-4.5"))
---

# 一、总览：各阶段的输入输出

|阶段|主要目标|典型输入|典型输出|
|---|---|---|---|
|0. 数据准备|构造训练语料与配套标注|原始文本/代码/图像/音频、过滤规则、去重规则|清洗后的 token 序列、样本对、偏好数据|
|1. 预训练|学会“预测下一个 token”|大规模无标签 token 序列|**Base model**（基础模型）|
|2. 监督微调（SFT）|学会按指令回答、形成基本行为|prompt-response 示范数据|**SFT model / instruct model 初版**|
|3. 偏好建模|学会区分“哪个回答更好”|同一 prompt 的多候选回答 + 排序偏好|**Reward model / preference model** 或隐式偏好目标|
|4. 偏好优化 / RL|让模型朝“更有用、更安全、更稳健”方向更新|SFT 模型、偏好模型/比较信号、交互 prompt|**Aligned model / post-trained model**|
|5. 安全与专项后处理|强化拒答边界、工具使用、多模态、风格控制|安全数据、工具轨迹、专项任务数据|**可发布模型**|

---

# 二、分阶段说明

## 0. 数据准备（Data pipeline）

### 主要工作

这是整个训练中最“工程化”的阶段，通常包括：

- 数据收集：网页、书籍、论文、代码、对话、多模态数据
    
- 清洗：去广告、去乱码、语言识别、格式标准化
    
- 过滤：低质量、有害、违法、重复、泄漏测试集内容
    
- 去重：文档级、段落级、近重复去重
    
- 切分与 tokenization：把原始内容转成 token
    
- 构造监督/偏好数据：
    
    - 指令-回答示范数据
        
    - 同 prompt 多回答排序数据
        
    - 安全红队数据
        
    - 工具调用轨迹数据
        

### 输入

- 原始互联网语料、代码库、文档、对话日志、多模态原始数据
    
- 过滤器、质量规则、标注规范
    

### 输出

- 预训练语料：`x1, x2, ..., xn` 的 token 序列
    
- SFT 数据：`(prompt, ideal_response)`
    
- 偏好数据：`(prompt, response_a, response_b, preference)`
    
- 安全/专项数据：拒答、工具调用、结构化输出等样本
    

### 本质

把“世界原始内容”转成模型可学的数据分布。

---

## 1. 预训练（Pre-training）

### 主要工作

这是基础能力来源。目标通常是**自回归语言建模**：给定前文，预测下一个 token。  
GPT-4.5 的公开表述仍强调扩大 **unsupervised learning**；这说明前沿模型的知识、语言、风格、泛化、部分推理能力，首先来自这一阶段。([OpenAI](https://openai.com/index/introducing-gpt-4-5/?utm_source=chatgpt.com "Introducing GPT-4.5"))

### 输入

- 大规模无标签 token 序列  
    形式上类似：  
    `["The", "capital", "of", "France", "is", ...]`
    

### 训练目标

- 最小化 next-token prediction loss
    
- 多模态模型则可能还包含跨模态预测/对齐目标，但语言主干仍常围绕 token 预测展开
    

### 输出

- **Base model**
    
- 特征：
    
    - 懂语言、世界知识、代码模式、统计关联
        
    - 但**不一定会很好地遵循用户意图**
        
    - 也不一定安全、稳健、会拒答
        

### 你可以把它理解成

“会说话、会续写、懂很多，但不一定会按要求做事”的原始基础模型。

---

## 2. 监督微调（SFT, Supervised Fine-Tuning）

### 主要工作

让 base model 从“会续写”变成“会按指令输出”。  
InstructGPT 的公开流程中，先收集 **labeler-written demonstrations**，再对 GPT-3 做 supervised fine-tuning。([arXiv](https://arxiv.org/abs/2203.02155?utm_source=chatgpt.com "Training language models to follow instructions with human feedback"))

### 输入

- `prompt`
    
- 人类或高质量教师模型给出的 `ideal_response`
    

例如：

```text
输入: “总结这篇文章的核心观点”
输出: “1. … 2. … 3. …”
```

### 训练目标

- 直接最小化模型输出与示范答案之间的监督损失（token-level cross-entropy）
    

### 输出

- **SFT model**
    
- 特征：
    
    - 更会聊天
        
    - 更能遵循 instruction
        
    - 输出格式更稳定
        
    - 初步形成“助手行为”
        

### 局限

- 它学到的是“模仿示范答案”
    
- 但对“两个都看起来合理的回答，哪个更好”这种细粒度偏好，还不够强
    

---

## 3. 偏好建模（Preference modeling / Reward modeling）

### 主要工作

这一阶段不是直接训练最终模型回答，而是先训练一个“评委”。

InstructGPT 路线里，先收集同一 prompt 下多个候选回答的人类排序，再训练 **reward model**。([OpenAI](https://openai.com/index/instruction-following/?utm_source=chatgpt.com "Aligning language models to follow instructions"))

### 输入

典型输入是：

- 一个 `prompt`
    
- 两个或多个候选回答 `response_a`, `response_b`
    
- 人类或 AI 给出的偏好：`a > b`
    

即：

```text
(prompt, chosen, rejected)
```

### 训练目标

- 学出一个评分函数，判断哪个回答更符合偏好：
    
    - 更有帮助
        
    - 更真实
        
    - 更安全
        
    - 更简洁/更符合风格
        

### 输出

- **Reward model / preference model**
    
- 或者在 DPO 一类方法中，不单独显式产出 reward model，而是把偏好关系直接写进优化目标
    

### 本质

把“人更喜欢什么样的回答”转成可优化的信号。

---

## 4. 偏好优化 / 强化学习（RLHF / RLAIF / 高算力RL / DPO类）

### 先纠正一个常见误解

并不是所有 SOTA 模型后训练都必须用传统 PPO 式 RL。  
但“用偏好信号继续优化模型”仍是主流。OpenAI 公开提到一些模型 post-training 包含 **SFT + high-compute RL**；Anthropic 公开的 Constitutional AI 也分为 supervised phase 和 reinforcement learning phase，只是偏好来源可由 AI feedback 提供。([OpenAI](https://openai.com/index/introducing-gpt-oss/?utm_source=chatgpt.com "Introducing gpt-oss"))

### 主要路线

#### 路线 A：RLHF

- Human preference → reward model
    
- 再用 RL 优化策略模型
    

#### 路线 B：RLAIF / Constitutional AI

Anthropic 的公开描述是：

- 先让模型按“宪法原则”自我批评与修订
    
- 再在 RL 阶段用 AI 反馈选择更无害输出，再训练偏好模型并做 RL。([anthropic.com](https://www.anthropic.com/research/constitutional-ai-harmlessness-from-ai-feedback?utm_source=chatgpt.com "Constitutional AI: Harmlessness from AI Feedback"))
    

#### 路线 C：DPO / ORPO / IPO 等

- 不显式训练或不依赖单独 reward model
    
- 直接用 chosen vs rejected 对进行偏好优化
    

### 输入

- SFT 模型
    
- prompt 集合
    
- 偏好数据 / reward model / AI feedback / constitution
    
- 可能还有工具使用成功率、格式正确率、代码通过率等可程序化奖励
    

### 输出

- **Aligned model / post-trained model**
    
- 相比 SFT：
    
    - 更符合偏好
        
    - 更会拒绝危险请求
        
    - 更稳
        
    - 更会长链任务、工具使用、复杂推理
        

### 本质

SFT 解决“会不会答”，  
偏好优化解决“答得好不好”。

---

## 5. 安全与专项能力后处理（Safety + capability post-training）

这一层在公开资料里常被并入“post-training”，但工程上经常可单独看。

### 主要工作

- 安全对齐：危险请求拒答、越狱鲁棒性、隐私、版权边界
    
- 工具使用训练：搜索、代码执行、函数调用、检索
    
- 结构化输出训练：JSON、schema、agent 轨迹
    
- 多模态对齐：图文问答、语音交互
    
- 长上下文与检索配合训练
    
- 风格控制：简洁、审慎、帮助性、可引用
    

OpenAI 公开资料提到 post-training 目标包括与模型规范对齐，并教会模型 **tool use** 和 reasoning。([OpenAI](https://openai.com/index/introducing-gpt-oss/?utm_source=chatgpt.com "Introducing gpt-oss"))

### 输入

- 安全对抗样本
    
- 红队数据
    
- 工具调用轨迹
    
- 结构化输出样本
    
- 多模态示范数据
    
- 专项 benchmark 反馈
    

### 输出

- **可部署模型**
    
- 常见附带系统：
    
    - policy model / refusal behavior
        
    - tool router
        
    - safety classifier / guard model
        
    - moderation pipeline
        

### 注意

严格说，这里有些能力并不一定全部写进主模型参数里，也可能部分由外部系统实现。

# 三、把整个流程串起来看：输入输出链条

可以抽象成下面这条流水线：

## 阶段 A：预训练数据

**输入**：原始互联网/代码/文档/多模态内容  
**输出**：清洗后的 token 语料

## 阶段 B：预训练

**输入**：token 语料  
**输出**：`Base Model`

## 阶段 C：SFT 数据构造

**输入**：prompt、人工示范、高质量教师输出  
**输出**：`(prompt, response)` 数据集

## 阶段 D：SFT

**输入**：`Base Model + (prompt, response)`  
**输出**：`SFT Model`

## 阶段 E：偏好数据构造

**输入**：同一 prompt 的多个候选回答 + 人类/AI 排序  
**输出**：`(prompt, chosen, rejected)` 数据集

## 阶段 F：偏好优化

**输入**：`SFT Model + preference data / reward model`  
**输出**：`Aligned / Post-trained Model`

## 阶段 G：安全与专项增强

**输入**：安全、工具、多模态、格式控制数据  
**输出**：`Deployable Assistant Model`

# 四、每个阶段解决什么问题

## 1. 预训练解决

- 语言能力
    
- 世界知识
    
- 代码模式
    
- 泛化能力

- 更贴近真实用户偏好
    

## 4. 安全与专项后处理解决

- 风险控制
    
- 工具调用
    
- 结构化输出
    
- 多模态可用性
    
- 发布可控性

# 五、最简版本结论

如果只保留最核心的主干，当前 SOTA 大语言模型训练可概括为：

## 两大部分

1. **Pre-training**
    
2. **Post-training**
    

## 四个核心阶段

1. **数据准备**  
    输入：原始语料  
    输出：清洗后的训练数据
    
2. **预训练**  
    输入：无标签 token 序列  
    输出：Base model
    
3. **监督微调（SFT）**  
    输入：指令-回答示范数据  
    输出：会遵循指令的 SFT model
    
4. **偏好优化（RLHF/RLAIF/DPO类）**  
    输入：偏好比较数据、reward/preference 信号  
    输出：对齐后的 assistant model
    

## 常见附加阶段

5. **安全与专项能力增强**  
    输入：安全、工具、多模态、格式化数据  
    输出：可部署产品模型

# 六、一个更直观的类比

- **预训练**：让模型“读完图书馆和互联网”，学会语言和知识
    
- **SFT**：教它“怎么像助理一样答题”
    
- **偏好优化**：教它“什么答案更好、更安全、更符合人类偏好”
    
- **专项后处理**：教它“如何调用工具、遵守规则、在产品里稳定工作”

# Prompt: 由base model到SFT model再到Post-trained model, 每步的结果改变网络结构吗? 还是仅通过修改模型参数实现? 

结论先说：
## 核心结论

**通常不改主干网络结构，主要是继续更新同一套模型参数。**  
也就是从 **base model → SFT model → post-trained model**，在主流做法里，通常是**同一个 backbone/架构**，通过不同阶段的数据和目标函数，持续修改参数得到新版本。InstructGPT 明确是从预训练 GPT-3 出发，先做 SFT，再用 PPO 继续微调策略模型；Meta Llama 3 也公开区分 **pre-trained** 和 **post-trained** 版本，但仍是同一 405B language model 的不同训练阶段版本。([arXiv](https://arxiv.org/abs/2203.02155?utm_source=chatgpt.com "Training language models to follow instructions with ..."))

---

## 一、base → SFT：通常是否改结构

### 通常情况

**不改结构。**  
做法一般是：

- 保持 Transformer 主体不变
    
- 保持层数、hidden size、attention 结构不变
    
- 用指令数据继续训练
    
- 更新权重参数，得到 SFT model
    

这本质上仍是“在原模型上继续梯度下降”。InstructGPT 的公开流程就是先从预训练模型开始做 supervised fine-tuning，而不是先换一个新网络结构。([arXiv](https://arxiv.org/abs/2203.02155?utm_source=chatgpt.com "Training language models to follow instructions with ..."))

### 输入输出角度看

- 输入：`base model 参数 + SFT 数据`
    
- 输出：`SFT model 参数`
    

因此这一步的变化，**主要是参数值变了，不是网络拓扑变了**。

---

## 二、SFT → post-trained：通常是否改结构

### 通常情况

**也通常不改主干结构，仍以参数更新为主。**  
例如 RLHF / 高算力 RL / DPO 类方法，核心都是在 SFT model 基础上继续优化策略模型，使其更符合偏好、更会推理、更会工具使用。OpenAI 对 gpt-oss 的公开表述就是 post-training 包含 **SFT stage + high-compute RL stage**，目标是教会模型 reasoning 和 tool use。这里强调的是训练阶段和目标，不是更换主干架构。([OpenAI](https://openai.com/index/introducing-gpt-oss/?utm_source=chatgpt.com "Introducing gpt-oss"))

### 这一步常见变化

- 主模型参数继续更新
    
- 输出分布、行为模式、拒答边界、格式稳定性发生明显变化
    
- 但 backbone 通常还是同一个
    

所以从结果上看，post-trained model 常常“像另一个模型”，但从工程上看，很多时候只是**同一网络结构下的一组新参数**。([OpenAI](https://openai.com/index/introducing-gpt-oss/?utm_source=chatgpt.com "Introducing gpt-oss"))

---

## 三、为什么很多人会误以为“改了结构”

因为后训练阶段经常会**新增周边组件**，但这不等于改了主模型主干结构。

### 典型新增组件

1. **Reward model / preference model**  
    InstructGPT 里有单独的 reward model，用来给回答打分；它是额外训练的模型，不是把主模型结构改掉。([arXiv](https://arxiv.org/pdf/2203.02155?utm_source=chatgpt.com "arXiv:2203.02155v1 [cs.CL] 4 Mar 2022"))
    
2. **Safety / guard model**  
    例如 Meta 公开了 Llama Guard 3，作为独立安全模型发布。它是额外模块，不是“把 Llama 主干改结构”。([arXiv](https://arxiv.org/abs/2407.21783?utm_source=chatgpt.com "[2407.21783] The Llama 3 Herd of Models"))
    
3. **工具调用系统 / 外部路由器**  
    模型学会 tool use，产品里常伴随 schema、router、executor、safety checker 等外部系统；这属于**系统层扩展**，不必然意味着 LLM backbone 结构改变。OpenAI 对 gpt-oss 的表述提到 post-training 教模型 tool use，但没有说因此重构主干。([OpenAI](https://openai.com/index/introducing-gpt-oss/?utm_source=chatgpt.com "Introducing gpt-oss"))
    

---

## 四、什么时候“可能会改结构”

有，但这通常不是你问的这条主线里的默认情况。

### 可能改结构的场景

#### 1. 多模态扩展

给纯文本模型加视觉编码器、跨模态投影层、音频模块。  
这时结构通常会新增模块。Meta 在 Llama 3 论文里明确提到通过 compositional approach 集成 image/video/speech 能力，这类就不只是纯参数更新。([arXiv](https://arxiv.org/abs/2407.21783?utm_source=chatgpt.com "[2407.21783] The Llama 3 Herd of Models"))

#### 2. 参数高效微调（PEFT）

例如 LoRA / adapter。  
这时主干参数可能冻结，只新增小型适配层。严格说这是**结构上加了附加模块**，但主干 backbone 没改。

#### 3. 词表或嵌入层调整

若 tokenizer 或 special tokens 发生变化，embedding/output head 维度可能随之调整。  
这属于局部结构变化，不是整体 backbone 重构。

#### 4. 长上下文或位置编码方案调整

例如 RoPE scaling、插值、额外位置参数。  
这有时算轻量结构/配置变化，但通常不是“从 base 到 SFT/post-training”的标准必需步骤。

#### 5. 真正换模型代际

如果从 GPT-4 到 GPT-4.5，或从 Llama 3 到 Llama 4，这往往可能涉及新的架构设计。  
但这已经不是“同一个 base model 经 SFT/post-training”的问题，而是**新模型训练**。

---

## 五、可以这样理解三者关系

### 1. Base model

- **结构**：定义好的 Transformer backbone
    
- **参数**：预训练后得到的一版权重
    

### 2. SFT model

- **结构**：通常与 base model 相同
    
- **参数**：在 base 参数基础上继续更新后的新权重
    

### 3. Post-trained model

- **结构**：通常仍与 SFT model 相同
    
- **参数**：在 SFT 参数基础上进一步优化后的权重
    
- **系统层**：可能同时挂上 reward model、guard model、tool router 等外部组件
    

---

## 六、最准确的一句话

**在同一模型族内部，base → SFT → post-trained 通常是“同一主干结构，不同参数快照”；真正常变的是参数、训练目标和外部配套模块，而不是 Transformer 主体拓扑。**

---

## 七、简表

|转换|主干结构是否通常改变|主要变化|
|---|---|---|
|Base → SFT|通常不变|用指令数据更新参数|
|SFT → Post-trained|通常不变|用偏好优化/RL/DPO等继续更新参数|
|Post-trained + 产品化|主模型通常不变，但可能外挂新模块|reward model、安全模型、工具系统、路由器|

# 一、最简答案

如果把 **o1 之类推理模型** 放回你前面那张图里，可以理解为：

## 普通 LLM 主线

**预训练**  
→ 学语言、知识、代码、模式  
→ 得到 **base model**

**SFT**  
→ 学会按指令回答  
→ 得到 **SFT model**

**强化的 post-training（核心区别）**  
→ 不是只学“答案长什么样”，而是学“如何分解问题、尝试不同策略、检查错误、再给答案”  
→ 得到 **reasoning post-trained model**

也就是说，**推理模型通常不是替代上述流程，而是在 post-training 阶段做得更重、更深、更强调 RL 和过程性推理能力**。([OpenAI](https://openai.com/index/learning-to-reason-with-llms/?utm_source=chatgpt.com "Learning to reason with LLMs"))

# 二、o1 这类模型与普通聊天模型的关键差别在哪里

## 1. 预训练阶段仍然存在

OpenAI 对 o1-mini 的表述很直接：  
“大语言模型如 o1 先在海量文本上 pre-trained”，而 o1-mini 则是在 pretraining 时更偏向 STEM reasoning，再经过与 o1 相同的高算力 RL pipeline。说明 reasoning model 仍然有普通 LLM 的预训练底座。([OpenAI](https://openai.com/index/openai-o1-mini-advancing-cost-efficient-reasoning/?utm_source=chatgpt.com "OpenAI o1-mini"))

### 这一阶段学到什么

- 语言能力
    
- 世界知识
    
- 数学/代码模式
    
- 基础泛化能力
    

### 输出

- 一个还不会“稳定深度思考”的 **base model**
    

---

## 2. SFT 阶段通常也仍然存在

OpenAI 对后续 reasoning 模型公开提到其 post-training 包含 **SFT stage + high-compute RL stage**。这说明 reasoning model 并不是跳过 SFT，而是仍先让模型具备基本 instruction-following 和答题习惯。([OpenAI 开发者](https://developers.openai.com/api/docs/guides/rft-use-cases/?utm_source=chatgpt.com "Reinforcement fine-tuning use cases | OpenAI API"))

### 这一阶段学到什么

- 按指令输出
    
- 规范格式
    
- 基本任务执行方式
    
- 初步示范式推理风格
    

### 输出

- 一个**会按要求答题**，但未必最擅长复杂推理的 **SFT model**
    

---

## 3. 真正拉开差距的是 reasoning-oriented post-training

这一步是核心。

OpenAI 公开说法有几条非常关键：

- o1 系列“trained with large-scale reinforcement learning to reason using chain-of-thought” ([OpenAI](https://openai.com/index/openai-o1-system-card/?utm_source=chatgpt.com "OpenAI o1 System Card"))
    
- o1 的性能会随着更多 **reinforcement learning** 和更多“thinking time”提升。([OpenAI](https://openai.com/index/learning-to-reason-with-llms/?utm_source=chatgpt.com "Learning to reason with LLMs"))
    
- o3/o4-mini 通过训练学会“refine their thinking process, try different strategies, and recognize their mistakes”，并被训练去推理何时、如何使用工具。([OpenAI](https://cdn.openai.com/pdf/2221c875-02dc-4789-800b-e7758f3722c1/o3-and-o4-mini-system-card.pdf?utm_source=chatgpt.com "OpenAI o3 and o4-mini System Card"))
    

这说明 reasoning model 的核心不是“多输出几步思维链”这么简单，而是：

- 学会把难题分解成中间步骤
    
- 学会尝试多种策略
    
- 学会发现并修正错误
    
- 学会在内部多花 token 进行推理
    
- 学会何时调用工具辅助求解

# 三、这一步到底怎么训练：可公开确认的抽象流程

不能把未披露细节当事实，但从官方公开内容，可以较可靠地抽象成下面流程：

## 阶段 A：先有一个较强的 base / SFT 模型

输入：

- 预训练权重
    
- SFT 数据
    

输出：

- 基础可用的 instruction model
    

## 阶段 B：构造“可验证或可评估”的推理任务

reasoning/RL 要发挥作用，通常需要某种奖励信号。公开资料没有完整列出 o1 的全部奖励设计，但从 OpenAI 对 reasoning RL、RFT 和 system card 的表述可以确认：  
这类训练依赖能区分“更好推理结果”的反馈，尤其适合数学、代码、科学、工具调用等**有较明确对错或质量信号**的任务。([OpenAI](https://openai.com/index/learning-to-reason-with-llms/?utm_source=chatgpt.com "Learning to reason with LLMs"))

典型输入可能包括：

- 难题 prompt
    
- 参考答案或可验证答案
    
- 单元测试 / 代码执行结果
    
- 工具调用是否成功
    
- 偏好或过程质量信号
    
- 安全规范文本
    

## 阶段 C：用强化学习让模型学会“更好的思考过程”

这里的目标不是只奖励最终表面措辞，而是让模型逐渐形成更有效的内部求解策略。  
OpenAI 明确说 o1 是用大规模 RL 教会模型“how to think productively using its chain of thought”。([OpenAI](https://openai.com/index/learning-to-reason-with-llms/?utm_source=chatgpt.com "Learning to reason with LLMs"))

### 这一步模型被鼓励什么

- 延迟作答，先想再答
    
- 拆解问题
    
- 检查中间结论
    
- 回溯错误
    
- 对复杂任务使用更多 test-time compute
    
- 在需要时使用工具
    

### 输出

- **reasoning post-trained model**

普通 post-training 更像：

- 回答更自然
    
- 更 helpful
    
- 更 harmless
    
- 更像助理
    

reasoning post-training 更像：

- 在复杂任务上形成内部搜索/反思/修正能力
    
- 学会把更多算力花在“想”上，而不是直接输出

OpenAI 对 o1/o3 系列的公开描述都指向这一点。([OpenAI](https://openai.com/index/learning-to-reason-with-llms/?utm_source=chatgpt.com "Learning to reason with LLMs"))

- reasoning tokens
    
- reasoning effort 控制
    
- 工具使用接口

# Prompt: test-time compute是什么? 具体如何工作? 为什么有时候我在使用5.4thinking模型更长的推理时间反而得到了一个比较差的结果? 

先校正前提：

## 前提修正

“更长的推理时间一定更好”这个前提不成立。  
OpenAI 的公开文档明确写到：

- `reasoning.effort` 控制模型在作答前生成多少 **reasoning tokens**；更高 effort 只是倾向于“更完整的推理”，不是保证结果单调变好。([OpenAI 开发者](https://developers.openai.com/api/docs/guides/reasoning/ "Reasoning models | OpenAI API"))
    
- 对 GPT-5.4，官方专门提醒：**higher effort is not always better**，合适的 effort 取决于任务形状，而不是直觉；并建议先固定 effort 做评测，再逐步调。([OpenAI 开发者](https://developers.openai.com/api/docs/guides/prompt-guidance/ "Prompt guidance for GPT-5.4 | OpenAI API"))
    
- 对日常任务，官方建议把 effort 设为 `minimal` 或 `low`，把更重的 effort 留给真正复杂的问题。([OpenAI 开发者](https://developers.openai.com/cookbook/examples/gpt-5/gpt-5_troubleshooting_guide/ "GPT-5 Troubleshooting Guide"))
    

所以，你观察到“5.4thinking 想更久反而更差”，从公开资料看，**这是正常可能发生的现象，不违背官方描述**。([OpenAI 开发者](https://developers.openai.com/cookbook/examples/gpt-5/gpt-5_troubleshooting_guide/ "GPT-5 Troubleshooting Guide"))

# 一、test-time compute 是什么

## 定义

**test-time compute** 指模型在“推理/回答阶段”消耗的计算量。  
对应到 reasoning model，主要就是模型在输出最终可见答案前，先花多少内部计算去“想”。OpenAI 对 o1 的公开表述是：性能会随着更多 **train-time compute** 和更多 **time spent thinking (test-time compute)** 提升。([OpenAI](https://openai.com/index/learning-to-reason-with-llms/?utm_source=chatgpt.com "Learning to reason with LLMs"))

## 和 train-time compute 的区别

- **train-time compute**：训练时花的算力，用来改参数
    
- **test-time compute**：部署后每次回答时花的算力，不改参数，只改变这次回答的推理深度
    

---

# 二、它具体如何工作

## 1. 以 reasoning tokens 的形式工作

OpenAI API 文档明确说明：reasoning models 在输入 token 和输出 token 之外，还会生成 **reasoning tokens**。这些 token 用来“think”，即分解问题、考虑多种方法，然后再产出最终可见答案。([OpenAI 开发者](https://developers.openai.com/api/docs/guides/reasoning/ "Reasoning models | OpenAI API"))

## 2. reasoning tokens 不直接显示给你

文档写得很清楚：

- 模型先生成 reasoning tokens
    
- 然后生成最终可见 completion
    
- 原始 reasoning tokens 不直接暴露
    
- 这些 reasoning tokens 之后会从上下文中丢弃，不继续带到下一轮对话里  
    但它们仍然占用本轮上下文窗口，并计入 token 使用量。([OpenAI 开发者](https://developers.openai.com/api/docs/guides/reasoning/ "Reasoning models | OpenAI API"))
    

## 3. `reasoning.effort` 本质上是在调“想多久”

官方文档写明：

- `reasoning.effort` 决定模型在回答前生成多少 reasoning tokens
    
- 支持 `none / minimal / low / medium / high / xhigh`（具体取决于模型）
    
- 更高 effort 往往意味着更多 latency 和 token 消耗。([OpenAI 开发者](https://developers.openai.com/api/docs/guides/reasoning/ "Reasoning models | OpenAI API"))
    

## 4. 在 ChatGPT 里，你看到的“thinking 更久”

公开资料没有把 ChatGPT 前端的全部内部实现细节都公开，但 GPT-5 介绍页明确说，ChatGPT 里的 GPT-5 是一个统一系统：普通模型负责多数问题，**GPT-5 thinking** 负责更难的问题，外加一个 router 决定用哪种模式。([OpenAI](https://openai.com/index/introducing-gpt-5/ "Introducing GPT-5 | OpenAI"))

因此，在 ChatGPT 里“更长推理时间”大体上可理解为：

- 路由到了更深的 reasoning 模式，或
    
- 分配了更高的 reasoning effort / 更多内部推理 token，或
    
- 在更长的多步流程里做了额外检查、重试或规划。  
    这里前两点有公开依据；最后一点是基于官方对“long-running tasks / multi-step workflows / verification loops”的说明做的谨慎推断。([OpenAI](https://openai.com/index/introducing-gpt-5/ "Introducing GPT-5 | OpenAI"))

# 三、为什么更长推理有时反而更差

这部分要分成“公开确认”和“合理推断”。

## A. 公开确认的部分

### 1. 更高 effort 不是普遍更优

OpenAI 对 GPT-5.4 明确写了：  
**higher effort is not always better**。([OpenAI 开发者](https://developers.openai.com/api/docs/guides/prompt-guidance/ "Prompt guidance for GPT-5.4 | OpenAI API"))

### 2. 日常任务不应默认高 effort

官方 troubleshooting guide 建议：

- routine work 用 `minimal` 或 `low`
    
- heavier effort 留给 genuinely complex problems。([OpenAI 开发者](https://developers.openai.com/cookbook/examples/gpt-5/gpt-5_troubleshooting_guide/ "GPT-5 Troubleshooting Guide"))
    

### 3. 需要靠 evals 选择 effort，而不是靠感觉

官方文档建议：

- 从较低或现有效果开始
    
- 逐步增加
    
- 只在 evals 明确显示收益时再上更高 effort，特别是 `xhigh`。([OpenAI 开发者](https://developers.openai.com/api/docs/guides/reasoning/ "Reasoning models | OpenAI API"))
    

---

## B. 机制上的解释

下面这些是对公开文档的工程化解释，不是 OpenAI 明说的内部细节。

### 1. 任务太简单时，额外推理会引入“过度加工”

简单任务更适合直接映射：

- 抽取
    
- 分类
    
- 简短改写
    
- 明确格式化输出
    

对这类任务，更多 reasoning 可能让模型：

- 开始补充不需要的解释
    
- 过度泛化你的要求
    
- 自己引入额外假设
    
- 从“执行任务”偏到“讨论任务”
    

这和官方建议“日常任务用低 effort”是一致的。([OpenAI 开发者](https://developers.openai.com/cookbook/examples/gpt-5/gpt-5_troubleshooting_guide/ "GPT-5 Troubleshooting Guide"))

### 2. 额外思考会扩大错误搜索空间

推理不是线性流水线，而更像在候选思路空间里搜索。  
更长推理时间意味着模型可能：

- 尝试更多路径
    
- 做更多中间判断
    
- 在多个 plausible 解释之间切换
    

当任务本身歧义较大、目标不够明确时，更多搜索不一定逼近正确答案，反而可能偏离你真正要的输出。  
这也是为什么 GPT-5.4 的提示指南反复强调：**明确输出契约、完成标准、验证规则** 比单纯加大 effort 更重要。([OpenAI 开发者](https://developers.openai.com/api/docs/guides/prompt-guidance/ "Prompt guidance for GPT-5.4 | OpenAI API"))

### 3. 模型优化的是“总体训练目标”，不是你这道题的绝对正确率

更长推理并不是“外接一个严格证明器”，它仍是模型按训练出来的策略在生成内部 reasoning tokens。  
因此它可能在总体平均上受益，但在某个具体问题上：

- 更啰嗦
    
- 更自信地走错路
    
- 更偏向训练中更常见的解法
    
- 更强调安全/格式/完整性而牺牲直接性
    

所以提升可以是**统计意义上的**，不是逐题单调。这个结论与官方“需要做 evals 决定 effort”相吻合。([OpenAI 开发者](https://developers.openai.com/api/docs/guides/reasoning/ "Reasoning models | OpenAI API"))

### 4. 更高 effort 可能把 token 预算花在“想”上，而不是“答”

官方文档说明 reasoning tokens：

- 占用上下文窗口
    
- 计入 output tokens
    
- 如果触发 `max_output_tokens` 或上下文限制，甚至可能在还没产生可见输出前就被截断。([OpenAI 开发者](https://developers.openai.com/api/docs/guides/reasoning/ "Reasoning models | OpenAI API"))
    

因此，在预算紧、上下文很长、任务又复杂时，更长推理可能导致：

- 真正答案空间变少
    
- 输出被截断
    
- 中后段回答质量下降
    

### 5. 长推理对提示质量更敏感

官方 GPT-5.4 指南强调，效果提升很大程度来自：

- 明确 output contract
    
- 明确 verification loop
    
- 明确 completion criteria  
    而不是盲目增加 effort。([OpenAI 开发者](https://developers.openai.com/api/docs/guides/prompt-guidance/ "Prompt guidance for GPT-5.4 | OpenAI API"))
    

也就是说，**高 effort + 模糊提示**，有时会比 **低 effort + 明确任务定义** 更差。

# 四、为什么你在 5.4thinking 上会体感“想更久但更差”

结合公开资料，最可能是以下几类情况：

## 1. 任务不需要深推理

例如：

- 简单事实问答
    
- 直接提取
    
- 简单翻译
    
- 明确格式转换
    

这类任务高 effort 常见问题：

- 多余展开
    
- 自作主张补背景
    
- 偏离原问题
    

官方已经明确建议这类任务优先低 effort。([OpenAI 开发者](https://developers.openai.com/cookbook/examples/gpt-5/gpt-5_troubleshooting_guide/ "GPT-5 Troubleshooting Guide"))

## 2. 问题目标不够封闭

例如你只说“分析一下”“深入思考”，但没有定义：

- 判断标准
    
- 输出格式
    
- 是否允许假设
    
- 不确定时该怎么处理
    

这时更长推理可能不是在逼近正确答案，而是在放大任务歧义。GPT-5.4 官方提示指南明确强调：要定义 output contract 和 done condition。([OpenAI 开发者](https://developers.openai.com/api/docs/guides/prompt-guidance/ "Prompt guidance for GPT-5.4 | OpenAI API"))

## 3. 你关心的是“直接性”，模型优化的是“完整性”

高 effort 往往更像：

- 多检查一步
    
- 多覆盖一个角落
    
- 多做一层解释
    

如果你的主观评价标准是：

- 更短
    
- 更直达结论
    
- 更少铺垫
    

那你会明显感觉“想久了反而更差”。这并不一定表示逻辑更差，而是**优化目标不匹配**。官方文档也把“完整 reasoning”和“速度/效率”明确视为取舍。([OpenAI 开发者](https://developers.openai.com/api/docs/guides/reasoning/ "Reasoning models | OpenAI API"))

## 4. 高 effort 并不保证更稳定

官方没有承诺“effort 越高结果越稳定”。相反，文档要求用户自行做 evals 决定。([OpenAI 开发者](https://developers.openai.com/api/docs/guides/reasoning/ "Reasoning models | OpenAI API"))  
这意味着：

- 某些题型高 effort 提升大
    
- 某些题型无明显收益
    
- 某些题型甚至回退

# 五、你可以怎么用，才更接近“长推理有收益”

## 适合提高 effort 的任务

- 数学证明/多步推导
    
- 复杂代码修复
    
- 多约束规划
    
- 长文档交叉验证
    
- 需要引用证据的综合研究
    
- 含多工具、多阶段依赖的任务  
    这些都和官方对 reasoning model、GPT-5.4 适用场景的描述一致。([OpenAI 开发者](https://developers.openai.com/api/docs/guides/reasoning/ "Reasoning models | OpenAI API"))
    

## 不适合盲目提高 effort 的任务

- 简单问答
    
- 结构化抽取
    
- 改写/润色
    
- 分类/路由
    
- 明确模板填充  
    官方对 routine work 的建议就是低 effort。([OpenAI 开发者](https://developers.openai.com/cookbook/examples/gpt-5/gpt-5_troubleshooting_guide/ "GPT-5 Troubleshooting Guide"))
    

## 比“多想一会儿”更重要的三件事

1. **明确输出契约**：你到底要表格、要结论、要步骤，还是要原文摘录
    
2. **明确停止条件**：做到哪一步算完成
    
3. **明确验证要求**：是否需要自查、引用、计算复核  
    这三点都直接对应 GPT-5.4 提示指南。([OpenAI 开发者](https://developers.openai.com/api/docs/guides/prompt-guidance/ "Prompt guidance for GPT-5.4 | OpenAI API"))

# 六、一个简化类比

把模型看成一个考生：

- **低 test-time compute**：先想到最可能答案就写
    
- **高 test-time compute**：先列几种思路、检查、比较、再作答
    

为什么高 compute 有时更差？

- 题太简单，第二种做法纯属绕远
    
- 题目表述不清，想得越多越容易走偏
    
- 时间花在纠结细节，而不是抓主干
    
- 多检查几轮后，反而把原本正确直觉改坏
    

这和人类考试里的“overthinking”是同一种现象，但这里更准确地说，是**推理预算与任务形状不匹配**。

# 七、最简结论

## test-time compute 是什么

是模型在回答时额外投入的计算量；在 reasoning model 中，主要体现为生成更多 **reasoning tokens** 来先想后答。([OpenAI 开发者](https://developers.openai.com/api/docs/guides/reasoning/ "Reasoning models | OpenAI API"))

## 它如何工作

模型在最终答案前，先生成内部 reasoning tokens；这些 token 不直接展示，但占用上下文并计费，随后被丢弃，不带入下一轮。([OpenAI 开发者](https://developers.openai.com/api/docs/guides/reasoning/ "Reasoning models | OpenAI API"))

## 为什么更长推理有时更差

因为：

- 高 effort 不是普遍更优
    
- 简单任务会被过度加工
    
- 模糊任务会被放大歧义
    
- token 预算可能被“想”消耗
    
- 模型优化的是统计平均表现，不是每题单调提升  
    这些都与 OpenAI 官方“higher effort is not always better”“routine work 用低 effort”“靠 evals 选择 effort”的公开指引一致。([OpenAI 开发者](https://developers.openai.com/cookbook/examples/gpt-5/gpt-5_troubleshooting_guide/ "GPT-5 Troubleshooting Guide"))
    

如果你愿意，我下一条可以把它进一步整理成一张表：  
**“哪些任务该用低/中/高 thinking，哪些任务 thinking 越高越容易变差”。**
